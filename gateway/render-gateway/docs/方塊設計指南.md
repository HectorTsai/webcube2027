# 🧱 方塊設計指南（AI 2026 Headless 完全體專用提示詞）

> **【重要】** 本文件是全站方塊（Cube）JSON 圖紙的最高指導原則。AI 在生成、修改任何方塊 JSON 時，必須嚴格遵守以下 Schema、架構分工與禁令，否則會導致前端渲染崩潰或 DOM 結構污染。

---

## 1. 核心哲學：單一節點與數據驅動

本系統的 `方塊.tsx` 是一個 Headless 晶片底座。它追求「零垃圾 Wrapper 節點」。
一個 JSON 定義在遞迴解析到最底層時，只會渲染出一個對應的原生 HTML 標籤。所有的排版、五態、顏色，全部透過屬性供電與 UnoCSS 類名驅動。

> **【2026-07 重大重構】方塊.tsx 純調度器原則**：`方塊.tsx` 僅負責按固定波段調度十條微型管線，不再包含任何跨管線的業務邏輯。所有 Alpine 狀態、樣式計算、結構展開、HTML 屬性映射均由專屬管線處理。如需支援新的方塊或功能，必須透過 JSON 定義或 fallbackRegistry 元件實現，禁止直接修改 `方塊.tsx`。

## 2. 統一方塊 JSON Schema

```typescript
{
  // ── 1. 定義與繼承 ──
  from: string;                         // 來源指針。可以是原生標籤 ("div", "ul", "a") 或 DB 方塊 ID ("方塊:方塊:容器")
  tag?: string;                          // ⭐ 標籤覆寫（肉身）。最終留在 HTML 上的標籤（例如 <footer>、<section>）

  // ── 2. 樣式與供電 ──
  className?: string;                   // 基礎 CSS 類（限 UnoCSS 原子類）。不可寫死會變動的樣式！
  style?: Record<string, string>;       // 靜態行內樣式
  styleConditions?: Record<string, string>; // 條件 CSS（UnoCSS 類名）。Key 必須對應一個 Boolean 類型的引數，Value 為 UnoCSS 原子類名字串（非 CSS 語法），直接拼入 className。

  // ── 3. 參數對外開放 (引數) ──
  args?: Record<string, {
    type: "string" | "number" | "boolean" | "array" | "object";
    description?: string;
    default?: unknown;
    options?: string[];
    variants?: Record<string, {          // 變體選項切換
      className?: string;                // 激活此選項時，追加入外層的類名
      style?: Record<string, string>;
      alpine?: object;
      on?: Record<string, string>;
      data?: Record<string, string>;
      [containerProp: string]: unknown;  // ⭐ 任意 key 會合併到 mergedArgs，傳遞給下游（如 rounded, padding）
    }>;
  }>;

  // ── 4. 參數內部鎖定 (預設值) ──
  defaults?: Record<string, unknown>;    // ⚠️ 內部鎖定的參數（如佈局、圓角）。不可與 args 的 key 重疊！

  // ── 5. 內容與插槽 ──
  children?: (object | string)[];        // 子方塊定義陣列，或純文字字串
  prepend?: PrependAppendItem[];         // 條件式前置內容（含 if 判定）
  append?: PrependAppendItem[];          // 條件式後置內容（含 if 判定）
  slots?: Record<string, {               // 具名插槽
    from: string;
    shareChildren?: boolean;
    children?: (object | string)[];
  }>;

  // ── 6. 結構包裹與互動 ──
  wrap?: { from: string; className?: string; style?: Record<string, string> }; // 群組穿衣包裹機制
  alpine?: { init?: string; bind?: Record<string, string>; attrs?: Record<string, string>; model?: string };
  on?: Record<string, string>;           // { "click": "open = true" } -> x-on:click
  data?: Record<string, string>;         // { "id": "{item.id}" } -> data-id="..."
  repeat?: string;                       // 陣列展開語法，例如 "{items}"
  mergedArgs?: Record<string, string>;   // 共享變數注入，支援內部 @api 呼叫（詳見 §8）
}

interface PrependAppendItem {
  if?: string;                           // 條件 Key（當引數為 falsy 時跳過渲染）
  from: string;
  className?: string;
  children?: (object | string)[];
  [key: string]: unknown;
}
```

## 3. AI 必知之四大核心機制與連動

### Mechanism 3.1：標籤覆寫（from 靈魂 vs tag 肉身）

- **from 決定靈魂（屬性繼承）**：決定這個方塊要載入哪張圖紙、繼承哪些變體（variants）與 defaults。
- **tag 決定肉身（最終 HTML 標籤）**：當你想讓一個方塊完全繼承 `"方塊:方塊:容器"` 的所有排版、圓角與陰影超能力，但最終在網頁上必須輸出為 `<footer>` 標籤以符合 SEO 語義時，必須使用 `tag` 進行覆寫。

### Mechanism 3.2：智慧五態供電（active、hover、selected、focus 與 activeStateName）

底層晶片內建了智慧五態判定，支援三種層級：

**層級 A：靜態 Boolean（純 CSS 狀態）**

當你傳入靜態 Boolean（如 `active: true`），它會渲染出 `data-active="true"`，由 UnoCSS 的 `[data-active]` 選擇器接管供電。

```json
{ "active": true, "hover": true }
```

**層級 B：Alpine 表達式（進階動態綁定）**

當你傳入運算式字串（含 `$` 或 `.`），它會智慧切換為 Alpine 動態綁定 `:data-active="$store.Menu.isOpen"`。

```json
{ "active": "$store.Menu.isOpen" }
```

**層級 C：activeStateName 宣告式動態切換（推薦，方塊內建機制）**

只需宣告 `activeStateName`，系統全自動生成三條 Alpine 指令：

| 自動生成 | 效果 |
|---|---|
| `x-bind:data-active` | 動態綁定 `data-active` 屬性，含 `?.` + `??` fallback 防呆 |
| `x-bind:style` | 斷電時自動注入 neutral 色 CSS 變數，覆蓋原色 |
| `x-init` | 防呆初始化 `Alpine.store('Container', {})` 及對應屬性 |

使用範例：

```json
{
  "args": {
    "activeStateName": {
      "type": "string",
      "description": "Alpine store 屬性名稱（用於動態切換）",
      "default": ""
    }
  }
}
```

呼叫端：

```tsx
// 簡單模式：僅傳屬性名，自動前綴 $store.Container
<Cube from="方塊:方塊:容器" activeStateName="btn_active_toggle" active hover />

// 完整路徑模式：直接指定 Alpine 表示式
<Cube from="方塊:方塊:容器" activeStateName="$store.Nav.isOpen" active hover />
```

對應的 Alpine 操控端：

```html
<button x-on:click="$store.Container.btn_active_toggle = !$store.Container.btn_active_toggle">
  點擊切換
</button>
```

> **AI 鐵律**：
> - `activeStateName` 與傳統的 `active: "$store.xxx"` 表達式**互斥**——前者生成 `x-bind:` 指令 + `x-init` + `x-bind:style`，後者僅生成 `:data-active`。
> - `activeStateName` 的路徑自動判別：不含 `$` 或 `.` 時自動前綴 `$store.Container.`；含 `$` 或 `.` 時直接使用。
> - 這兩個屬性會與佈景主題系統的「風格（Style）」模組聯動（利用 `[data-active]` 選擇器供電），嚴禁在 `className` 裡手動拼裝狀態類名（如 `active ? 'bg-blue-500' : 'bg-gray-200'`）。

### Mechanism 3.3：全自動顏色供電（color）

本系統擁有全自動的 UnoCSS 顏色映射。

> **AI 鐵律**：只要在引數中傳入 `color: "primary"`（支援 `primary`, `secondary`, `accent`, `neutral`, `info`, `success`, `warning`, `error`, `ghost`），底層會自動注入 `cube-color-{color}` 類名，並自動啟用佈景主題的 OKLCH 變數。嚴禁自創 `text-[#ff0000]` 或手動拼寫 `--c-current` 行內樣式。
>
> **current utility 透明度支援**：`text-current`、`text-current-content`、`bg-current`、`bg-current-content`、`border-current` 五個核心 utility 均支援 `/{opacity}` 修飾符（如 `bg-current/10`、`bg-current-content/10`）。其中 `bg-current-content` 使用 `--c-current-content` CSS 變數（反襯色，非黑即白），適合斑馬紋等需與底色反襯的場景。

### Mechanism 3.3a：顏色繼承模式（`"current"` variant）

當子方塊需要**自動繼承父層方塊的顏色**（而非使用自己的品牌色）時，使用 `"current"` variant 模式。此模式完全依賴 CSS 自訂屬性（`--c-current` / `--c-current-content`）的自然繼承，無需任何 JS 介入。

#### 使用方式

在 `args.color.variants` 中新增 `"current"` 選項，並將 `default` 設為 `"current"`：

```json
{
  "args": {
    "color": {
      "type": "string",
      "default": "current",
      "options": ["current", "primary", "secondary", "accent", "base", "neutral"],
      "variants": {
        "current":   { "className": "text-current-content" },
        "primary":   { "className": "text-primary" },
        "secondary": { "className": "text-secondary" }
      }
    }
  }
}
```

#### 運作原理

1. **父層**透過 `cube-color-{color}`（如 `cube-color-primary`）注入 CSS 自訂屬性到 DOM
2. **子方塊**的 `default: "current"` → wrapper 套上 `text-current-content`（只設 `color`，不設 `--c-current*`）
3. `stylePipeline` 會自動注入 `cube-color-current`（`--color-current-raw` 不存在 → CSS 自訂屬性無效 → 瀏覽器自動 fallback 到繼承值）
4. 子方塊內部元素使用 `border-color: currentColor` 或 `text-current-content` 時，會自動取到父層穿透下來的反襯色

#### 設計者自由度

`"current"` 的 `className` 完全由方塊設計者決定，可依需求選擇：

| 需求 | className |
|------|-----------|
| 文字跟隨父層主色 | `text-current` |
| 文字跟隨父層反襯色 | `text-current-content` |
| 背景跟隨父層主色 | `bg-current` |
| 邊框跟隨父層反襯色 | `border-current-content` |
| 多重組合 | `text-current-content bg-current/10` |

> **AI 鐵律**：若方塊會被巢狀使用（如分隔線放在列表內、圖示放在按鈕內），必須在 `color` arg 中提供 `"current"` 選項並設為 `default`。這確保子方塊在未指定 `color` 時自動繼承父層顏色，而非使用自己的品牌色 default。

### Mechanism 3.4：群組包裹穿衣（wrap）

當你需要製作一個列表、下拉選單或表格時，使用 `wrap` 屬性。底層晶片會自動進行語義推導：

- 當外層 `wrap.from` 為 `"ul"` 或 `"ol"` 時，內層的子項（children）會被自動包裹上 `<li>` 標籤，並套用斑馬紋電路。
- 當外層 `wrap.from` 為 `"select"` 時，內層子項會被自動包裹上 `<option>` 標籤。
- 其餘情況預設包裹 `<div>`。

> **AI 鐵律**：此時 `prepend` 與 `append` 可以完美繞過衣服規範，直接塞入如「列表標題」等非 `li` 結構。

### Mechanism 3.5：三波段平行加速架構（2026-07 重大重構）

方塊渲染引擎採用三波段執行模型，最大化利用並行計算：

**Wave 1（串行依賴）** — 必須按順序執行，後續管線依賴其結果：
| 順序 | 管線 | 職責 |
|------|------|------|
| ① | chainPipeline | 解析 `from` 引用，合併父層定義與當前定義 |
| ② | rbacPipeline | 角色權限控制熔斷站（目前全 Pass） |
| ③ | apiPipeline | 解析 `@api` 內部 API 呼叫，注入 `mergedArgs` |
| ④ | rawParserPipeline | 解析 `{"raw": "..."}` 原始 HTML 標記 |

**Wave 2（單獨）** — 單一管線，需等待 Wave 1 完成：
| 順序 | 管線 | 職責 |
|------|------|------|
| ⑤ | argsPipeline | 合併 defaults、args 預設值、使用者傳入值，生成 `localArgs` |
| - | Phase 1 變體傳播 | `傳播變體值()` — 將 variant 非標準鍵注入 `localArgs` |

**Wave 3（並行）** — Promise.all 同時執行，互不依賴：
| 順序 | 管線 | 職責 |
|------|------|------|
| ⑥ | alpinePipeline | 解析智慧五態、Alpine 綁定、動態 disabled 聯動 |
| ⑦ | stylePipeline | Phase 2 樣式計算，合併 `definition.className` 與 `localArgs.className` |
| ⑧ | renderPipeline | 子節點遞迴渲染、$$CHILDREN$$ 注入、結構鍵模板插值 |
| ⑨ | htmlPropsPipeline | 生成純 HTML 屬性（src、alt、width、height 等） |

> **AI 鐵律**：
> - `className` 來源：`definition.className`（區塊級固定類名）+ `localArgs.className`（使用者動態傳入），由 `stylePipeline` 合併。
> - 動態 disabled 聯動：`alpinePipeline` 檢測 `x-bind:disabled` / `:disabled`，自動生成對應的 `x-bind:data-active` / `x-bind:data-hover`。
> - 結構鍵模板插值：子定義中結構鍵欄位（如 `className: "{className}"`）的模板字串，由 `renderPipeline` 用父層 `localArgs` 插值後傳遞給子方塊。

### Mechanism 3.6：子項分配巨集（`$$CHILDREN` 系列）

當一個方塊的 `children` 陣列包含多個子方塊定義，且需要將外部傳入的 JSX 子項分配到不同位置時，使用 `$$CHILDREN` 巨集。

#### 可用巨集一覽

| 巨集 | 含義 | 範例場景 |
|------|------|---------|
| `$$CHILDREN$$` | 全部 JSX 子項 | body 區塊接收所有內容 |
| `$$FIRST_CHILD$$` | 第 1 個子項 | media 區塊取圖片 |
| `$$FIRST_IMAGE_CHILD$$` | 第 1 個**圖片**子項（找不到則空） | 卡片 media 區塊，無圖時自動透明 |
| `$$REST_CHILDREN$$` | 第 1 個以後的全部；若與 `$$FIRST_IMAGE_CHILD$$` 搭配，則為扣除該圖片的其餘所有 | body 區塊取文字內容 |
| `$$LAST_CHILD$$` | 最後 1 個子項 | footer 區塊取按鈕 |
| `$$CHILD_N$$` | 第 N 個子項（0-based） | `$$CHILD_2$$` = 第 3 個 |
| `$$CHILDREN_N_M$$` | 第 N 到 M 個（不含 M） | `$$CHILDREN_0_2$$` = 前 2 個 |
| `$$CHILDREN_N_END$$` | 第 N 個到結尾 | `$$CHILDREN_2_END$$` = 第 3 個起 |

#### 用法

巨集寫在 `children` 陣列中子物件的 `children` 欄位：

```json
{
  "from": "方塊:方塊:容器",
  "children": [
    { "from": "方塊:方塊:div", "className": "media", "children": "$$FIRST_IMAGE_CHILD$$" },
    { "from": "方塊:方塊:div", "className": "body",  "children": "$$REST_CHILDREN$$" }
  ]
}
```

呼叫端傳入 3 個 JSX 子項（含圖片）：

```tsx
<Card>
  <img src="..." />           ← 找到圖片 → $$FIRST_IMAGE_CHILD$$ → media div
  <h2>標題</h2>               ← $$REST_CHILDREN$$ → body div
   <p>內容</p>                 ← $$REST_CHILDREN$$ → body div
</Card>
```

呼叫端傳入純文字（無圖片）：

```tsx
<Card>
  <h2>純內容卡片</h2>         ← 找不到圖片 → media div 空
  <p>說明文字</p>             ← $$REST_CHILDREN$$ → body div（全部內容）
</Card>
```

#### 設計原則

- **語義自明**：巨集名稱應讓人不查文件也能讀懂意圖。`$$REST_CHILDREN$$` 優於 `$$CHILDREN_1_END$$`
- **一個子項只出現一次**：`$$FIRST_CHILD$$` + `$$REST_CHILDREN$$` 確保每個子項只分配到一個位置
- **無子項時安全降級**：沒有外部傳入的 JSX 子項時，巨集展開為空陣列，不會報錯

> **AI 鐵律**：設計方塊時，只有當你真的需要把不同 JSX 子項分配到不同 DOM 位置時才用巨集。如果所有子項進同一個容器，直接用 `$$CHILDREN$$`（或省略 children 欄位，系統預設全部注入）。

## 4. 內部鎖定（defaults）vs 對外開放（args）

AI 設計 JSON 時，必須嚴格區分這兩者的邊界：

- **args**：使用者在後台或呼叫端可以傳入、切換的參數（如 `color`, `size`, `active`）。
  - args 的「預設值」寫在 arg 定義內部的 `default` 欄位。
- **defaults**：此元件傳給 `from` 繼承物件的**內定值**，強制鎖定、不容許外部覆寫（如頁尾永遠 `border: "none"`、永遠 `align: "center"`）。
  - `defaults` **不是「預設值」**，是寫死的內部固定值。兩者概念完全不同。

> ⚠️ **致命規則**：只要 key 進了 `defaults`，該 key **絕對不能再出現於 `args`**。因為 `defaults` 意味著「不給使用者選擇」，那就不該在 `args` 中暴露。

```json
// ✅ 正確：頁尾不讓使用者選 border / shadow / align / justify
{
  "from": "方塊:方塊:容器",
  "defaults": {
    "border": "none",
    "shadow": "none",
    "align": "center",
    "justify": "center"
  },
  "args": {
    "color": { "type": "string", "default": "base" },
    "padding": { "type": "string", "default": "xs" }
    // ← 沒有 border / shadow / align / justify，因為已被 defaults 鎖定
  }
}
```

## 5. 變體（Variant）設計的絕對原則

當容器的幾何特徵（如圓角 `rounded`、內距 `padding`）需要跟隨某個引數（如 `size`）變化時，必須透過變體屬性（Variant Prop）直接傳遞，絕對不要拼寫在 `className` 字串中。

### ❌ 錯誤示範

AI 亂編的 className，會與佈景主題幾何 Token 打架：

```json
{
  "variants": {
    "sm": { "className": "rounded-sm p-2 w-32" },
    "lg": { "className": "rounded-lg p-6 w-48" }
  }
}
```

### ✅ 正確示範

利用變體屬性穿透，完美與佈景主題系統無縫接軌：

```json
{
  "args": {
    "size": {
      "type": "string",
      "default": "md",
      "options": ["sm", "md", "lg"],
      "variants": {
        "sm": { "rounded": "sm", "padding": "xs", "width": "32" },
        "md": { "rounded": "md", "padding": "sm", "width": "40" },
        "lg": { "rounded": "lg", "padding": "md", "width": "48" }
      }
    }
  }
}
```

### 5.1 Arg 繼承與 Variant 自動繼承

當子方塊透過 `from` 繼承父方塊時，如果子方塊要**開放父方塊已有的 arg**（例如容器的 `direction`），必須遵守以下規則：

#### 規則一：子方塊必須在 `args` 中宣告該 arg（可發現性）

即使該 arg 的 `variants` 由父方塊定義，子方塊仍需在自己的 `args` 中宣告該 arg，讓使用者與 AI 知道此方塊支援哪些參數。`type`、`default`、`description`、`options` 可以照常填寫。

```json
// ✅ 正確：列表繼承容器的 direction，宣告 arg 但不寫 variants
{
  "from": "方塊:方塊:容器",
  "args": {
    "direction": {
      "type": "string",
      "description": "排列方向（繼承自容器）",
      "default": "col",
      "options": ["col", "row"]
      // ← 不寫 variants，系統自動繼承容器的 direction variants
    }
  }
}
```

#### 規則二：不要重複撰寫 variants（避免混淆）

子方塊宣告 arg 時，**不要**重複撰寫 `variants`。系統在 `處理方塊引用` 階段會自動從父方塊繼承對應 arg 的 variants 定義。

重複撰寫 variants 會導致以下問題：
- 與父方塊的 variant 定義不一致，造成視覺混亂
- 維護負擔倍增：父方塊改動 variant 時，所有子方塊都需同步更新
- AI 容易在 variant 中塞入不該有的 class，繞過佈景主題系統

#### 規則三：只有當你要「覆蓋」父的 variant 時，才需要寫

如果子方塊確實需要與父方塊**不同**的行為（例如某個 variant 要改成不同的 className），此時才需要在子方塊的 arg 中明確寫出 `variants`，主動覆蓋父的定義。

```json
// ✅ 正確：子方塊主動覆蓋父的 direction variants（row 改為 row-reverse）
{
  "from": "方塊:方塊:容器",
  "args": {
    "direction": {
      "type": "string",
      "default": "col",
      "options": ["col", "row"],
      "variants": {
        "col": { "className": "flex-col" },
        "row": { "className": "flex-row-reverse" }
      }
    }
  }
}
```

#### 底層實作原理

合併發生在 `處理方塊引用` 階段（`components/方塊.tsx`）：

```
1. args = { ...parentArgs, ...childArgs }      // shallow merge，子覆蓋父
2. 遍歷合併後的 args：
   若 key 存在於 childArgs 且 merged[key] 無 variants 且 parentArgs[key] 有 variants
   → 自動補上 parentArgs[key].variants
```

> **AI 鐵律**：子方塊宣告繼承來的 arg 時，一律不寫 `variants`。需要覆蓋才寫。自動繼承機制保證 variant 來源唯一，杜絕父/子 variant 不一致的 bug。

### 5.2 變體傳播兩階段處理（2026-06 重構）

**底層實作**：方塊.tsx 在處理 variant 時採用兩階段傳播：

**Phase 1 — 非標準鍵搶先傳播**：
```
遍歷 args 的 variants：
  若 variant 中有非標準鍵（非 className/style/on/data/alpine）
  → 立即將該 key-value 合併到 mergedArgs
  → 讓下游 children 或其他 arg 處理時可立即取用
```

**Phase 2 — 標準樣式收集**：
```
再次遍歷 args 的 variants：
  僅收集 className / style / on / data / alpine 五個標準鍵
  → 拼入最終 className、綁定事件等
```

此兩階段設計解決了「子方塊 variant 定義了 `padding`/`rounded`/`fontSize` 等非標準鍵，但因為容器 arg 早已處理完畢，導致這些參數無法傳遞到渲染結果」的系統性 Bug。

> **AI 鐵律**：設計 variant 時，非標準鍵（如 `padding`、`rounded`、`fontSize`、`width`）會自動傳播到 `mergedArgs`，無需手動處理。variant 中的 className 應只包含該選項特有的樣式（如 `flex-col` 或 `flex-row`），幾何類的 padding/rounded 應使用非標準鍵傳播。

## 6. 常用禁令與黃金法則速查表

| ❌ 嚴禁行為 | ✅ 正確做法 |
|---|---|
| 方塊的 `from` 等於自己的 id | 一律引導至 `"from": "方塊:方塊:容器"`（僅容器/圖示/圖片除外） |
| 擅自拼寫 `text-[#ff0000]` 或 `bg-blue-500` | 使用全自動 `color` 參數，或使用全站 preset 原子類 |
| 在 `className` 裡用三元運算子寫死狀態樣式 | 傳遞 `active` / `hover` 引數，讓佈景主題的「風格」模組接管 |
| 在 `styleConditions` 中使用 CSS 語法（如 `li:nth-child(even){...}`）或 `<style>` 標籤 | `styleConditions` 只能輸出 UnoCSS 原子類名字串（如 `"divide-y divide-current/15"`），直接拼入 className |
| 在 `children` 中使用 `{ "$api": { "path": "..." } }` 動態抓取外部 API | 改用 `mergedArgs` 定義 `@api` 內部變數，children 中以 `{變數名}` 插值引用（詳見 §8） |
| 設計 `variant`（樣式變體）arg 來切換 solid/outline/ghost 等外觀 | 已廢棄。樣式變體已全面轉移至「風格（Style）」模組，由佈景主題統一管轄 |
| 呼叫複合方塊時忘記傳遞 `context` | 外層呼叫處一律綁定 `context={c}`，確保非同步 DB 載入鏈不中斷 |
| 在 `className` 中寫死 `bg-transparent` 覆蓋背景 | 背景由風格系統全權控制（active → `bg-current`，inactive → `bg-base-50`） |
| 同時使用 `cursor-not-allowed` 與 `pointer-events-none` | 兩者互斥：pointer-events-none 會讓游標穿透，無法顯示禁止符號 |
| 在 `alpine.init` 中使用單行注釋 `//` | 多層 init 會以 `; ` 串接（如 `foo(); // 注釋` + `bar()`），導致後續陳述式被誤吞為注釋。如需注釋請使用 `/* ... */` |

## 7. 斷電與 Disabled 狀態的系統級行為

當方塊的 `active={false}` 或 `disabled={true}` 時，`方塊.tsx` 會**全自動**執行以下防線，AI 不需手動處理：

| 自動行為 | 機制 | 適用範圍 |
|---------|------|---------|
| `cursor-pointer` → `!cursor-not-allowed` | 正則替換 className | 所有元素 |
| `aria-disabled="true"` | 注入無障礙標記 | 所有元素 |
| 移除 `href`、`action`、`formaction` | `原生點擊屬性` 清單 | 所有元素 |
| `data-active="false"` → inactive 風格 CSS | UnoCSS 規則 `c-style-apply` | 掛載 `c-style-apply` 的元素 |

> **AI 鐵律**：設計超連結類方塊時，不要寫死 `bg-transparent`（會蓋掉 inactive 灰底），不要自己加 `pointer-events-none`（會讓 `cursor-not-allowed` 失效）。這些全部由系統自動處理。

## 8. API 呼叫的雙軌機制：`$api`（禁止）vs `@api`（允許）

方塊系統提供兩種 API 呼叫方式，但用途與安全邊界截然不同：

### 8.1 `@api` 內部共享變數（`mergedArgs`）— ✅ 允許

在 `mergedArgs` 中使用 `@api/...` 規格字串，系統會在 `準備核心Args` 階段透過 `InnerAPI` 內部調用對應 API，並將結果注入 `mergedArgs`，供 `{變數名}` 插值使用。

**機制**：
```
mergedArgs → resolveMergedArgsDef() → InnerAPI(context, path) → data 注入 mergedArgs
```

**範例** — 頁尾方塊使用 `@api` 抓取年份、翻譯文字：

```json
{
  "from": "方塊:方塊:容器",
  "mergedArgs": {
    "currentYear": "@api/v1/temporal/fullYear",
    "prefix": "@api/v1/translate/copyright",
    "suffix": "@api/v1/translate/all rights reserved"
  },
  "children": [
    {
      "from": "span",
      "text": "{prefix} \u00a9 {copyright.開始年份}-{currentYear}"
    }
  ]
}
```

**解析流程**：
1. `resolveMergedArgsDef` 將 `@api/` 替換為 `/api/`，呼叫 `InnerAPI(context, "/api/v1/temporal/fullYear")`
2. 取回 `{ success: true, data: 2026 }`，將 `currentYear = 2026` 寫入 `mergedArgs`
3. `children` 中的 `{currentYear}` 由 `substitute()` 自動替換為 `2026`

> **鐵律**：`@api` 只能用在 `mergedArgs` 中，不可出現在 `children`、`slots` 或其他欄位。

### 8.2 `$api` 外部動態引用（`children` / `slots`）— ❌ 禁止

在 `children` 或 `slots` 中使用 `{ "$api": { "path": "...", "field": "..." } }` 語法，會在渲染階段動態呼叫外部 API。

**禁用原因**：
- 違反數據驅動原則：API 資料應在 `mergedArgs` 階段統一解析，不應散落在 children 定義中
- 破壞方塊的可預測性：外部 API 的副作用無法被 `defaults` 鎖定或 `args` 覆蓋
- 增加渲染鏈複雜度：每個 child 獨自發起 API 呼叫，難以除錯與效能優化

**正確做法**：將需要的資料定義在 `mergedArgs` 中，透過 `@api` 統一解析，children 中以 `{變數名}` 引用。
