# 開發流程書

請閱讀 `docs/規格書.md`，並按照以下階段執行開發任務。全程遵守規格書定義的命名規則（§3.J）、API 回應格式（§3.G）、日誌規範（§3.M）及**統一資料庫訪問架構（§3.B）**。

## 🚨 重要架構限制

**資料入口**：所有資料操作必須透過 `資料池` 全域單例 (`database/資料池.ts`)。

**🔧 分層權限限制**：

| 層級 | ✅ 允許操作 | ❌ 禁止操作 |
|------|------------|------------|
| **Middleware** | Context 操作、資料池 | 無 |
| **API 層** | Context 取得、資料池 | 直接資料庫連線 |
| **Service 層** | Context 取得、資料池 | 直接資料庫連線 |
| **Renderer 層** | API 調用 (`app.request` / `InnerAPI(c, path)`) | Context（僅傳入 InnerAPI）、Service、直接資料庫 |

**關鍵規則**：
- ✅ 所有服務層級都可直接 `import { 資料池 }` 操作資料（不再需要透過 Context 拿 DB）
- ✅ `System` getter 可直接拿 L2 adapter 做後台系統操作
- ✅ Renderer 層只能透過 API 調用獲取資料

**AI 編輯器注意**：必須遵循此分層架構限制！

# 第一階段 - 基礎設施與開發規範

對應規格書：§2 技術棧、§3.G API 回應格式、§3.J 命名規則、§3.M 日誌系統、§4 目錄結構

### 1. 專案初始化
- 建立 `deno.json`，配置 `imports`（包含 `hono`, `@unocss/preset-wind4`, SurrealDB client, `@dui/smartmultilingual` 等）。
- 建立 `.env` 檔案，配置 6 個環境變數：`SECRET_KEY`, `CACHE_ENABLED`, `AI_API_KEY`, `AI_PROVIDER`, `LOG_LEVEL`, `LOG_TO_FILE`。
- 建立專案目錄結構（依規格書 §4）：`/core`, `/database/models`, `/database/seeds`, `/middleware`, `/services/apiService`, `/services/mediaService`, `/services/rendererService`, `/components`, `/utils`。

#### 1.1 Deno 運行環境旗標（2026-06 更新）
- **Deno 2.7+ 起 `Temporal` API 已穩定**，不再需要 `--unstable-temporal` 旗標。
- `dev` / `start` 指令目前保留 `--unstable-cron`（cron 仍為 unstable）。
- 排程服務 `services/scheduler/index.ts` 內 `GCD轉表達式` 採用 `Temporal.Duration.round()` 自動進位：
  - 整分鐘（< 1h）→ `*/N * * * *`
  - 整小時 → `0 */N * * *`（自動避開 `*/60` 邊界錯誤）
  - 整天數 → `0 0 */N * *`
  - 非整除（90、150 分鐘）→ fallback `* * * * *`

#### 1.2 資料庫寫入規範（2026-06 更新）
- **Adapter 全量替換語意**：`database/adapter/sqlite.ts` 的 `更新()` 使用 `INSERT OR REPLACE`，意即**整筆覆蓋**。
- **呼叫端責任**：修改單一欄位時，呼叫端必須**先取完整 `toJSON()`，再覆蓋目標欄位**，最後傳入 adapter。
  - 範例：`database/資料池.ts` 的 `更新最後執行` → 先 `record.toJSON()` → `完整資料.最後執行 = 時間` → `創建或更新(完整資料)`
- **Adapter 防呆**：`更新()` 會自動偵測傳入物件是否為 Model 實例（`typeof toJSON === 'function'`），若是則自動呼叫確保完整性。

#### 1.3 AI JSON 解析順序規則（2026-06 更新）
- `utils/AI重試.ts` 的 `嘗試解析JSON` **必須先匹配物件 `{...}` 再匹配陣列 `[...]`**。
- 原因：AI 常回 `{"安全": true, "問題": []}`，若先匹配陣列，regex 會抓到內嵌的 `[]` 回傳空陣列，導致 `審查結果.安全` 永遠 `undefined`。
- 此規則適用於所有透過 `聊天並解析JSON()` 的 AI Task。

### 2. 日誌系統
- 建立 `utils/logger.ts` 統一日誌介面（依規格書 §3.M）。
- 支援 `LOG_LEVEL` 等級控制（debug / info / warn / error）及 `LOG_TO_FILE` 檔案輸出（寫入 `/logs` 目錄）。
- 全專案禁止直接使用 `console.log`，統一透過 Logger。

### 3. API 回應格式標準
- 建立統一的回應工具函式（依規格書 §3.G）：
    - 成功回應：`{ success: true, data, source }`
    - 錯誤回應：`{ success: false, error: { code, message, status } }`
- 定義錯誤 Code 常數：`NOT_FOUND`, `FORBIDDEN`, `DELETE_PROTECTED`, `VALIDATION_ERROR`, `BAD_REQUEST`, `INTERNAL_ERROR`。

### 4. L1 核心與資料池
- 建立 `database/資料池.ts` 全域單例：
  - L1 連線管理（SQLite）。
  - 動態掃描 `database/models/*.ts` 自動初始化 seed。
- 建立 `系統資訊` Model（含網站名稱、描述、商標等 SEO 欄位，依規格書 §3.H）。
- 建立 Seed 資料機制，初始化系統資訊預設值。

### 5. 啟動初始化與內部調用驗證
- 在 `main.ts` 初始化 Hono App。
- **啟動時資料庫初始化**：
  - `await 資料池.初始化()` — L1 連線 + 動態掃描 models/ 目錄自動 seed
  - `await 資料池.初始化L2()` — L2 連線（從 L1 讀取連線資訊）
  - `設定App(app)` — InnerAPI 內部路由依賴（模組層級注入，替代過去的 middleware `c.set('app')`）
- **錯誤處理**：使用 Hono 原生 `app.onError(...)`，不需要 middleware 包裝。
- **零 middleware**：移除所有 `app.use('*')` middleware，所有業務邏輯改為 lazy load 透過 InnerAPI。
- 建立測試路由 `GET /api/v1/info/system`，驗證系統資訊讀取。
- 實作內部調用測試：透過 InnerAPI 驗證 API-First 架構運作正常。

### 驗收條件
- 目錄結構完整建立。
- Logger 可正常輸出各等級日誌。
- API 回應統一格式。
- 資料池 L1/L2 初始化正常，SQLite seed 載入正常。
- 零 middleware，請求鏈無 `app.use('*')` 註冊。
- 內部調用流程正常。

# 第二階段 - 三層資料庫與核心 API

對應規格書：§2 Databases、§3.A 資料存取層、§3.D 多國語言、§3.K API 總覽

### 1. 資料庫分層初始化
- 在 `main.ts` 啟動時執行：
  - `await 資料池.初始化()` — L1（SQLite）連線 + 自動 seed。
  - `await 資料池.初始化L2()` — L2（SurrealDB）連線，連線資訊從 L1 讀取。
- **L3 懶載入**：`資料池` 的 `路由層級(host)` 方法內建偵測：
  - 若 host 不存在於連線池，自動觸發 `初始化L3(host)`。
  - 初始化失敗 → 該 host 跳過 L3，降級 L2 → L1。
  - 無需 middleware 或手動呼叫。

### 2. Model / Seed 系統
- 建立核心 Model（依規格書 §3.A，繁中命名）：
    - **佈景主題六模組**：`配色`、`骨架`、`風格`、`動畫`、`裝飾`、`圖示集`、`佈景主題`
    - **內容**：`頁面`、`方塊`
    - 所有 Model 包含共通欄位 `可刪除: boolean`。
    - 多國語言欄位使用 `MultilingualString` / `SmartContent`（依規格書 §3.D）。
- 建立對應的 Seed JSON 檔案，Seed 資料 `可刪除 = false`。
- `資料池.初始化()` 自動載入 models 目錄並初始化 Seed 資料（無需 middleware）。

### 3. 核心 CRUD API（單數命名 + 路徑參數）

> API 設計原則：所有資源檔案使用單數命名（`theme.ts`, `color.ts` 等），透過路徑參數區分操作（`/all` 取得列表、`/{id}` 指定資源）。

**統一 CRUD 模式**（六模組全適用）：

| 方法 | 路徑 | 用途 |
|------|------|------|
| `GET` | `/api/v1/{resource}` | 取得當前資源 |
| `GET` | `/api/v1/{resource}/all` | 取得所有（支援 `?include_seeds=false`） |
| `GET` | `/api/v1/{resource}/{id}` | 取得指定資源 |
| `POST` | `/api/v1/{resource}` | 建立新資源 |
| `PUT` | `/api/v1/{resource}/{id}` | 更新資源 |
| `DELETE` | `/api/v1/{resource}/{id}` | 刪除（檢查 `可刪除`） |

適用資源：`theme`, `color`, `skeleton`, `style`, `animation`, `ornament`, `icon-set`

**頁面與方塊 API**：
- `GET /api/v1/page/all`、`GET /api/v1/page/{id}`、`GET /api/v1/page/{id}/標題`（巢狀欄位）、`GET /api/v1/page/path/{path}`（巢狀路徑）、`POST`、`PUT`、`DELETE`
- `GET /api/v1/cube`（全部）、`GET /api/v1/cube/{id}`、`POST`、`PUT`、`DELETE`

**三層查詢架構**：資料池自動管理 L3 → L2 → L1 降級，保證預設值存在。

**系統 Seeds 保護**：
- DELETE 檢查 `可刪除` 欄位，系統 seeds (`可刪除: false` 且來自 L1) 無法刪除。
- 回傳 `SYSTEM_SEED_PROTECTED` 錯誤碼。

### 4. 預設值與資訊 API
- **預設值 API**：`GET /api/v1/defaults` / `GET /api/v1/defaults/{module}`（module: theme/color/skeleton/style/animation/ornament/icon-set）
- **統一資訊 API**：`GET /api/v1/info`（優先網站資訊，回退系統資訊）
- **系統/網站資訊**：`GET /api/v1/info/system`、`GET /api/v1/info/website`

### 5. UnoCSS 資源 API
- `GET /api/v1/unocss/classes`：取得自訂 preset 的可用 class 列表（供 AI 生成用，依規格書 §3.L）。

### 驗收條件
- 所有 CRUD API 正常運作，回應格式統一。
- 資料池三層查詢 fallback 機制正確。
- Seed 資料自動初始化（動態掃描 models/ 目錄）。
- DELETE 保護機制有效。
- host 解析正確（含埠號），不同網站隔離運作。

# 第三階段（已完成）— 渲染引擎、UnoCSS 與 Media Service

對應規格書：§2 Styling、§3.C 渲染管線、§3.F Media Service、§3.H SEO、§3.I 快取

### 1. UnoCSS Runtime 生成器
- 建立 UnoCSS 核心模組（`/unocss/` 下多檔案架構）：
    - `unocss/unocss.ts`：主入口，初始化 UnoCSS Generator（引入 `@unocss/preset-wind4`）。
    - `unocss/unocss-rules.ts`：自訂 rules（`c-style-apply`, `c-div-active`, `c-div-hover`, `c-div-inactive` 等），將風格模組的 active/inactive/hover class 映射到 `[data-active]` / `[data-hover]` 屬性選擇器。
    - `unocss/unocss-theme.ts`：自訂主題，學習 daisyUI 設計理念，使用 CSS Variables + oklch 色彩格式，根據亮度自動計算文字顏色（≥40% 黑，<40% 白）。
    - `unocss/unocss-shortcut.ts`：自訂 shortcuts（`btn`, `btn-primary`, `card`, `input` 等組合 class）。
    - `unocss/unocss-generator.ts`：UnoCSS 生成器類別，接收骨架/配色/風格/裝飾實例，動態產生完整 preset。
    - `unocss/unocss-cache.ts`：樣式快取管理。
    - `unocss/animate.ts`：組合式動畫 rules（`animate-in`, `fade-in`, `slide-in-from-bottom` 等）。
- 實作 `產生樣式(html: string)` 函式供 Renderer 調用。

### 2. 路由分發器
- 在 `main.ts` 實作 Hono 路由分發（不需 middleware）：
    - `/api/*` → apiService
    - `/medias/*` → mediaService
    - 其餘 → rendererService
    - 各 Service 自行處理錯誤及 404。
- **API 路由智能回退**：`apiService/index.ts` 從完整路徑一路退到底：
  - `/api/v1/skeleton/abc/def/ghi` → 嘗試 `skeleton/abc/def/ghi.ts` → `skeleton/abc/def.ts` → `skeleton/abc.ts` → `skeleton.ts`
  - 命中後，剩餘路徑設為 `params.id`（如 `abc/def/ghi`），由 handler 自行分割處理巢狀欄位。

### 3. Renderer Service
- 建立 `services/rendererService`：
    - 透過 `app.request` 或 `InnerAPI(c, path)`（傳入 Context 以節省資源）向 API 取得頁面資料、佈局方塊、內容陣列。
    - **方塊渲染**（統一 JSON 描述模式）：
        - 所有方塊以 JSON 格式描述（`from`, `className`, `style`, `args`, `children` 等）。
        - `from: "div"|"span"|...` → `Cube.tsx` 直接渲染為原生 HTML。
        - `from: "方塊:方塊:..."` → 透過 fallback registry 委派給對應元件（如 `Container.tsx`）。
        - 支援 JSX children 和資料驅動（seed/API）兩種等價用法。
        - children 支援巢狀遞迴（上限 10 層）。
        - **⚠️ children 內 arg 設計原則**：children 中的子方塊屬性會經過 `智慧插值` 自動替換 `{variable}` 模板（2026-07 新增）。但建議**不要在 children 內使用 `{variable}` 來傳遞與父級同名 arg**（如 `"size": "{size}"`），因為字串替換後會覆蓋父級繼承值。應直接省略該 arg，讓父級 mergedArgs 自然流到子方塊；若需不同值，則明確填入具體值（如 `"size": "lg"`）。
        - **三波段平行加速架構**（2026-07 重大重構）：方塊渲染引擎升級為三波段執行模型，`方塊.tsx` 回歸純調度器角色：
          - **Wave 1（串行依賴）**：① chainPipeline → ② rbacPipeline（角色權限控制，目前全 Pass）→ ③ apiPipeline → ④ rawParserPipeline
          - **Wave 2（單獨）**：⑤ argsPipeline → Phase 1 變體傳播（`傳播變體值()`）
          - **Wave 3（並行）**：Promise.all([⑥ alpinePipeline, ⑦ stylePipeline, ⑧ renderPipeline, ⑨ htmlPropsPipeline])
        - **stylePipeline 兩階段拆分**：Phase 1（`傳播變體值()`）在 Wave 3 之前執行，將 variant 非標準鍵注入 `localArgs`；Phase 2（`運行樣式與條件管線()`）為純計算，與其他管線並行。
        - **stylePipeline 合併 localArgs.className**：使用者透過 JSX prop 傳入的 `className` 不再丟失，與 `definition.className` 合併後輸出。
        - **alpinePipeline 動態 disabled 聯動**：檢測 `localArgs` 中的 `x-bind:disabled` / `:disabled`，自動生成對應的 `x-bind:data-active` / `x-bind:data-hover`，確保 Alpine 動態禁用時視覺同步變灰。
        - **方塊.tsx 純調度器**：移除所有跨管線的業務邏輯，僅負責按固定波段調度十條微型管線。所有 Alpine 狀態、樣式計算、結構展開、HTML 屬性映射均由專屬管線處理。
    - **佈景主題載入流程**：
        1. 從 API 取得當前佈景主題（六模組指針）。
        2. 逐一取得配色、骨架、風格、動畫、裝飾、圖示集的完整資料。
        3. CSS Variables 掛載到 `<html>` style 屬性（骨架的幾何 Token + 配色的 OKLCH 色彩）。
        4. 風格模組的 CSS 組合拳透過 UnoCSS rules（`c-style-apply` 等）在渲染時生效。
        5. 動畫模組的 class 映射在對應元件上動態套用。
    - 渲染完成後呼叫 `unocss.generate(html)` 產生 CSS，注入 `<head><style>`。
    - SEO meta tags 從網站資訊取得，fallback 到系統資訊（依規格書 §3.H）。
    - 404 時渲染錯誤頁面。

### 4. Media Service
- 建立 `services/mediaService`（依規格書 §3.F）：
    - 列表 API：`GET /medias/{type}` 支援分頁（page, pageSize）、排序（sort, order）、搜尋（search）。
    - 單筆 API：`GET /medias/{type}/:id` 回傳媒體本身。
    - 特殊路徑：`/medias/icon` 抓取網站/系統資訊的商標作為 favicon。
    - 各類型找不到時回傳對應的預設資源（非 JSON 錯誤）。
    - SVG 從 SurrealDB 讀取。

### 5. 語言 API
- 建立 `apiService/language.ts`：`GET /api/v1/language`
  - 從 Cookie (`lang`) → Accept-Language header → `InnerAPI(c, '/api/v1/info/system/預設語言')`（純字串欄位，不回觸過濾循環）→ fallback `zh-tw` 解析請求語言。
  - 解析後驗證 `InnerAPI(c, '/api/v1/info/website/語言')`（網站白名單），不符則取白名單第一項。
  - 回傳 `{ 語言: "zh-tw" }`。
  - 所有下游 API handler 透過 `取得語言(c)` 取得當前語言（享受 InnerAPI 快取）。

**語言 API 捷徑**（2026-06 新增）：
| 端點 | 用途 |
|------|------|
| `GET /api/v1/language/預設語言` | 捷徑 → 等同 `/api/v1/info/system/預設語言` |
| `GET /api/v1/language/可用語言` | 捷徑 → 等同 `/api/v1/info/website/語言` |

**巢狀欄位查詢支援**（2026-06 新增）：
- 六模組（skeleton/theme/color/style/ornament/animate）現支援 `/{resource}/{id}/{欄位路徑}`：
  - 先查詢完整記錄 → `資料過濾器.一般過濾()` → 穿透巢狀欄位
  - 支援多層 `/` 穿透（如 `/skeleton/骨架:骨架:經典/配置/radius-md`）
  - 不存在欄位回 404

**翻譯快取機制**（2026-06 更新）：
- `查詢翻譯快取` 改為遍歷匹配原文（`查詢所有列表`），不再用 hash ID（`查詢單一`）
- 原因：種子資料 ID 為手動命名（如 `單字:單字:copyright`），hash ID 永遠對不上

### 6. 快取機制
- **InnerAPI 透明快取**：`InnerAPI(c, path)` 內建 Context 層級快取，同一請求內重複路徑自動去重。呼叫端無需手動管理。
- **不再需要** `.env` 的 `CACHE_ENABLED` 開關（已被 InnerAPI 透明快取取代）。
- **禁止二次快取**：任何服務不得自行建立 Map 快取 API 回應，直接呼叫 `InnerAPI` 即可。

### 驗收條件
- 完整的頁面可在瀏覽器渲染，包含 CSS 與 CSS Variables。
- 佈景主題六模組正確載入並生效（骨架幾何 + 配色色彩 + 風格五態 + 動畫效果 + 圖示集鍵位映射）。
- Media Service 可回傳媒體或預設資源。
- favicon 路徑正常運作。
- SEO meta tags 正確注入。
- 同一請求內重複 InnerAPI 呼叫自動命中快取（可透過 `X-Cache: HIT-REQUEST-CONTEXT` header 驗證）。

### 7. AI 代碼審查排程 (L2 — 背景安全層)

  - 掃描所有 `已檢驗` 為空的方塊，逐一提交給 AI 審查。
  - 審查通過 → 計算完整性 hash 寫入 `已檢驗` 欄位。
  - 審查失敗 → 標記 `已檢驗: '審查失敗'`，記錄詳細原因，等待人工處理。
  - 透過 `database/models/排程記錄.ts` + `database/seeds/排程記錄.json` 管理排程任務（由 `services/scheduler/index.ts` 的 Deno.cron GCD 心跳驅動）。

### 8. Cube 渲染前驗證 (L3 — 渲染安全層)
- 在 `components/Cube.tsx` 實作渲染前 hash 驗證：

### 11. 工具註冊中心（AIToolRegistry，2026-07 新增）
- 建立 `utils/AIToolRegistry.ts`：全域單例 `ToolRegistry`，集中管理所有 AI 可調用的工具：
  - **`register(config)`**：註冊一個工具，接受 `名稱`、`描述`、`參數`（OpenAI function calling schema）、`必填`、`執行` 函數。
  - **`list()`**：回傳所有已註冊工具的 `ToolDefinition[]` 清單（供 prompt 組合器注入 AI）。
  - **`find(名稱)`**：依名稱尋找工具定義。
  - **`execute(名稱, 參數)`**：執行指定工具，自動捕捉錯誤回傳標準化結果。
  - **`數量`**：getter，回傳工具總數（用於判斷是否已初始化）。
- 設計原則：
  - 新增金剛模組只需 `register(tool定義)`，ThemeGenerator 零改動。
  - 工具定義使用 OpenAI function calling 格式（adapter 層統一標準）。
  - 每個 tool 自行處理錯誤、回傳 `Tool執行結果`（含 `工具名稱`、`成功`、`資料`、`錯誤`）。
- 建立 `services/aiService/task/tool-registry.ts`：初始化所有六大金剛生成工具：
  - `generate_color` → 配色生成器（ColorGenerator）
  - `generate_skeleton` → 骨架生成器（SkeletonGenerator）
  - `generate_style` → 風格生成器（StyleGenerator）
  - `generate_animate` → 動畫生成器（AnimateGenerator）
  - `generate_ornament` → 裝飾生成器（OrnamentGenerator）
  - `generate_icon_set` → 圖示集生成器（IconSetGenerator）
  - 每個工具的描述包含該模組的完整規格摘要（如配色含 9 色值說明），幫助 AI 精準判斷何時調用。
  - 擴充方式：新增第七金剛時，只需在此檔案 import 並新增對應的 register 呼叫。

### 12. 工具調用迴圈（AI工具調用.ts，2026-07 新增）
- 建立 `utils/AI工具調用.ts`：`工具調用生成()` 函數，實現 AI 工具調用 + 自我修正的完整迴圈：
  - 流程：
    ```
    呼叫 AI（含工具清單）
      → AI 回傳 tool_calls？
        → 是：執行工具 → 結果餵回對話歷史 → 重新呼叫 AI（最多 5 輪）
        → 否：解析 JSON → 驗證結構
          → 成功：回傳結果
          → 失敗：自我修正（同 自我修正生成 模式，最多 3 輪）
    ```
  - **工具調用階段**：支援同一輪並行執行多個 tool_call，執行結果以 `'tool'` 角色餵回對話歷史，格式為 `✅ 工具 "xxx" 執行成功。結果：{...}` 或 `❌ 工具 "xxx" 執行失敗。錯誤：...`。
  - **JSON 解析階段**：使用 `嘗試解析JSON()` 從 AI 回應中提取 JSON。
  - **結構驗證階段**：可傳入自訂 `驗證器` 函數驗證 JSON 結構，失敗時將具體錯誤餵回 AI 修正。
  - **參數控制**：`最大輪次`（預設 5）、`最大修正次數`（預設 3）、`初始溫度`（0.7）、`調整溫度`（0.3）。
  - 回傳結果包含：`json`、`原始回應`、`serverID`、`providerType`、`工具調用次數`、`紀錄`（工具調用歷史，供前端顯示）。

### 13. 自我修正生成迴圈（自我修正.ts，2026-07 新增）
- 建立 `utils/自我修正.ts`：`自我修正生成()` 函數，讓 AI 看到自己的錯誤並自動修正：
  - 流程：
    ```
    呼叫 LLM → 解析 JSON → 驗證結構
      → 成功：回傳結果
      → 失敗：把具體錯誤訊息加入對話歷史 → 重新呼叫 LLM（最多 N 輪）
    ```
  - **JSON 語法錯誤**：第一輪給詳細提示（引號、括號、逗號檢查），後續輪次給簡潔提示。
  - **結構驗證錯誤**：將具體錯誤描述（如「缺少必要欄位：名稱」）餵回 AI，讓 AI 根據錯誤修正。
  - **溫度調降**：首次使用 `初始溫度`（0.7），修正輪次使用 `修正溫度`（0.3）提高穩定性。
  - 回傳結果包含：`json`、`原始回應`、`serverID`、`providerType`、`修正次數`（0 = 首次成功）。
- 與 `AI工具調用.ts` 的關係：
  - `自我修正.ts`：純文字 JSON → 解析 → 驗證 → 錯誤修正（無工具調用）。
  - `AI工具調用.ts`：內建工具調用迴圈，最終輸出階段同樣使用自我修正模式。

### 14. 結構驗證器（結構驗證.ts，2026-07 新增）
- 建立 `utils/結構驗證.ts`：為各生成類型定義基礎結構驗證函數：
  - `驗證方塊結構(json)`：檢查 `from` 欄位存在、`children` 格式正確、無危險內容。
  - `驗證頁面結構(json)`：檢查 `路徑`、`標題`、`方塊列表` 格式。
  - `驗證風格結構(json)`：檢查五態 CSS 定義格式。
  - `驗證佈景主題結構(json)`：檢查六大金剛指針齊全、命名格式正確。
  - 驗證器簽名 `(json: unknown) => string | null`：回傳 `null` 代表通過，字串為人類可讀的錯誤描述。
  - 內建安全檢查：`<script>/<iframe>` 標籤封鎖、`javascript:` URI 攔截、inline event handlers 檢測、UnoCSS arbitrary value 禁令。

### 15. 提示詞組合器（提示詞組合器.ts，2026-07 新增）
- 建立 `utils/提示詞組合器.ts`：`組合提示詞()` 函數，將任務提示詞 + 設計規則 + 預取上下文組合為完整 AI 系統提示詞：
  - **LLM Prompt Caching 最佳化**：設計規則（最長固定前綴）→ 預取上下文（方塊清單/CSS classes，半固定）→ 任務提示詞（變動）→ 使用者訊息（每次不同）。利用 LLM provider 的前綴匹配機制自動 cache 設計規則，節省 50-70% 輸入 token。
  - **依生成類型精準分配清單**：
    | 生成類型 | 自動注入的上下文清單 |
    |----------|--------------------|
    | 方塊 | CSS classes |
    | 頁面 | 方塊清單 + CSS classes |
    | 風格 | CSS classes + 配色清單 |
    | 佈景主題 | CSS classes + 配色 + 骨架 + 風格 + 動畫 + 裝飾 + 圖示集（六大金剛） |
    | 配色/骨架/動畫/裝飾/圖示集 | 對應的既有模組清單 + CSS classes |
  - 支援 `生成類型` 參數，自動決定注入哪些預取清單。
  - 內建 60 秒模組級快取減少重複 API 調用。

### 16. 六大金剛子生成器（2026-07 新增）
- 佈景主題協調器 `theme-generator.ts` 為上層 Task，透過 AI 工具調用智能調度六大金剛子生成器，架構如下：
  ```
  ThemeGenerator（協調層）
    ├── 查詢六大金剛清單（自動注入 prompt）
    ├── AI 判斷：既有模組夠用 → 直接組合
    └── AI 判斷：既有模組不夠 → 調用 generate_xxx 工具
        ├── generate_color → ColorGenerator（配色生成器）
        ├── generate_skeleton → SkeletonGenerator（骨架生成器）
        ├── generate_style → StyleGenerator（風格生成器）
        ├── generate_animate → AnimateGenerator（動畫生成器）
        ├── generate_ornament → OrnamentGenerator（裝飾生成器）
        └── generate_icon_set → IconSetGenerator（圖示集生成器）
  ```
- 各子生成器實作規範：
  - **ColorGenerator** (`color-generator.ts`)：生成 OKLCH 格式 9 色值（主色/次色/強調色/中性色/背景/資訊/成功/警告/錯誤），支援色相角、亮度、彩度參數。
  - **SkeletonGenerator** (`skeleton-generator.ts`)：生成 CSS Token 七大族系（radius/spacing/font/leading/border/icon/image），支援幾何結構參數。
  - **StyleGenerator** (`style-generator.ts`)：生成五態 CSS 定義（active/inactive/hover/selected/focus），支援漸層、陰影、邊框參數。
  - **AnimateGenerator** (`animate-generator.ts`)：生成 17 個動畫場景的 Animate.css class 映射，支援動畫類型、速度、緩動參數。
  - **OrnamentGenerator** (`ornament-generator.ts`)：生成 8 個裝飾位置的 SVG 引用配置，支援裝飾風格參數。
  - **IconSetGenerator** (`icon-set-generator.ts`)：生成 14 個標準語義鍵位的圖示 ID 對照表，從既有圖示庫中選取，支援圖示風格參數。
- 每個子生成器使用 `自我修正生成()` 確保輸出 JSON 格式正確，並自動記錄使用日誌。
- 擴充方式：新增第七金剛時，只需在 `tool-registry.ts` 新增 register 呼叫，ThemeGenerator 完全不需改動。

- **工具調用迴圈正常**：AI 可主動調用 generate_xxx 工具生成新模組，工具結果正確餵回對話歷史。
- **自我修正機制正常**：AI 在 JSON 語法錯誤或結構驗證失敗時，能根據錯誤訊息修正輸出。

### 2. 權限警衛（pageService 層）
- POST / PUT / DELETE 需認證後才能存取（依角色區分系統管理員 / 網站管理員）。
- GET 保持開放（或依需求設定）。
- **警衛位置**：放在 `pageService` 渲染階段：
  - 渲染頁面前呼叫 `InnerAPI(c, '/api/v1/auth/check/{pageId}')` 檢查權限。
  - 無權限 → 直接回 404（不對外洩漏頁面存在性，攻擊者無法區分「不存在」與「無權限」）。
  - 有權限 → 繼續渲染流程。

### 3. 整合測試
- 端到端流程測試：
    - 系統管理員設定 L2 連線 → 網站管理員申請網站 → 設定佈景主題（六模組）→ AI 生成方塊 → 頁面渲染（含風格五態、動畫效果）。
- 效能測試：確認快取開啟後的回應時間改善。
- 安全測試：確認未認證的寫入請求被拒絕。

### 4. 生產環境準備
- `LOG_TO_FILE=true` 測試。
- 錯誤處理全面檢查（各 Service 的 404 / 500 處理）。

### 驗收條件
- 認證流程完整，權限控制有效。
- 完整端到端流程可執行。
- 生產環境配置就緒。

# 第六階段（待開發）— Client Transformer.js 整合

> **狀態：尚未開始。** 保留做為未來開發參考，避免遺漏。

### 背景
伺服器端已有 `TransformerProvider`（server transformer.js），並透過 seed 機制建立了預設的 transformer AI server 作為 fallback。
但客戶端 transformer.js 可以在請求送出前先攔截，減少伺服器負擔、降低成本、保護隱私。

### 待做事項

#### 1. 客戶端 Transformer.js 模組
- 建立 `/client/ai/transformer-client.ts` 或類似模組。
- 使用 `@xenova/transformers`（已加入 `deno.json` imports）。
- 在客戶端預載一個輕量模型（如 `Xenova/Qwen2.5-1.5B-Instruct`）。
- 對於簡單任務（翻譯、硬體查詢等），先嘗試本地推理，失敗才送 server。
- 需考量首次載入模型的時間（下載 + 初始化），可做背景預載或 Service Worker 快取。

#### 2. 客服預檢（方案 B — Transformer.js Embedding）
- 用 Transformer.js 跑 embedding 模型（如 `Xenova/bge-small-en-v1.5`）做語意相似度比對。
- 流程：
  1. 使用者提問 → 客戶端計算問題 embedding
  2. 與網站內容 embedding 做 cosine similarity
  3. 低於門檻（不相關）→ 直接拒絕，不送 server，零 token 消耗
  4. 高於門檻 → 送 server AI Pool 處理
- 網站內容 embedding 可預先計算並快取（內容變更時重算）。

#### 3. AI 小幫手客戶端整合
- `POST /api/v1/ai/assist/*` 四個端點，簡單任務優先走客戶端：
  - **硬體推測** (`/assist/hardware`) → 知識型任務，建議走 server（小模型不一定知道特定 CPU 規格）
  - **翻譯填空** (`/assist/translate`) → 優先走 client transformer.js（輕量、免費）
  - **內容建議** (`/assist/context`) → 品質需求高，走 server
  - **自訂提示** (`/assist`) → 依 complexity 判斷

#### 4. 客戶端 Fallback 鏈（與 server 對稱）
```
客戶端本地 transformer.js
  → 失敗或品質不足
    → Server AI Pool（網站自備 L3 → 系統收費 L2 → 系統免費 L2 → Server transformer）
```

#### 5. 優化考量
- 模型首次載入較慢，可考慮在頁面 idle 時背景預載。
- 翻譯等簡單任務可限制 maxTokens 提高速度。
- 客戶端 transformer.js 僅支援 CPU 推理（WASM），不適合重量任務。
