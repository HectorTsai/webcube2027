# data-gateway 規格書

> 最後更新：2026-07-22

---

## 1. 概述

data-gateway 是 WebCube2027 的**資料代理 Gateway**，負責統一管理 L2（SYSTEM）與 L3（TENANT）資料庫連線，並提供 RESTful API 讓前端與其他服務操作資料。

**核心職責**：

- 作為全域資料庫連線池（Pool）的唯一管理者
- 提供統一的資料 CRUD API（RESTful）
- 管理安裝流程（首次使用時設定 L2 資料庫、建立預設角色與管理員）
- 提供內部 API（Inner API）供 auth-gateway 驗證使用者

## 3. 啟動流程

```
main.ts
  │
  ├── setL1(gw.l1)                      ← 註冊 L1 單例（utils/l1.ts），供 middleware/handler 使用
  │
  ├── createGateway()                   ← @dui/framework
  │     ├── L1 init (l1.json)            ← 資料目錄: ./data/
  │     ├── Crypto key init (存入 L1)    ← registerKey() 註冊
  │     ├── Hono app 建立
  │     └── 檔案路由載入 (routes/)
  │
  ├── dataPool.setConfigStore(gw.l1)    ← 提供 L1 給 Pool
  │
  ├── 安裝檢查 Middleware               ← routes/_middleware.ts
  │     ├── 無 L1 l2_connection → 進入 /setup
  │     ├── 允許公開路徑（/setup、/css/、/images/、/favicon）
  │     └── 允許公開 API（/api/setup）
  │
  ├── JWT 中介層                        ← routes/api/_middleware.ts
  │     ├── 接受任何有效 JWT（匿名或已認證）
  │     └── 提取 tenant、jwt_type 存入 Context（供 handler 自行判斷）
  │
  ├── Inner API 註冊
  │
  ├── Data API 註冊
  │
  └── gw.start()                        ← 啟動 HTTP Server
```

### 3.1 使用 `@dui/framework`

data-gateway 使用 `@dui/framework` 的 `createGateway()` 統一處理以下初始化：

1. **L1 KV Store** — 在 `gateway/data-gateway/data/l1.json` 建立設定儲存
2. **加密金鑰** — 自動產生 256-bit 金鑰，存入 L1 的 `_crypto_key`，透過 `registerKey()` 註冊到 crypto 模組
3. **Hono 應用程式** — 建立 HTTP server
4. **檔案路由** — 自動載入 `routes/` 目錄下的 handler

### 3.2 資料池初始化

```typescript
dataPool.setConfigStore(gw.l1);  // 提供 L1 參考

// 若 L1 已有 L2 連線資訊，自動連接
if (await dataPool.config?.get('l2_connection')) {
  await dataPool.initL2();
}
```

## 4. 資料目錄

所有資料檔案儲存在各自的資料目錄下，確保獨立部署時路徑正確。

```
gateway/data-gateway/
└── data/
    ├── l1.json              ← L1 設定儲存
    │                          ├── _crypto_key（加密金鑰）
    │                          ├── l2_connection（L2 連線資訊，加密儲存）
    │                          └── auth_gateway_url（auth-gateway 服務 URL，安裝時設定）
    └── db.sqlite            ← L2 SQLite 資料庫（若選擇 SQLite 作為 L2）
```

> SQLite 作為 L2 時，檔案路徑自動解析為 `gateway/data-gateway/data/` 下。

## 5. API 路由

### 5.1 安裝 API

| 方法 | 路徑 | 說明 |
|------|------|------|
| POST | `/api/setup` | 首次安裝 — 設定 L2 連線、auth-gateway URL、建立管理員 |

請求主體：

```json
{
  "管理員帳號": "admin",
  "管理員密碼": "password",
  "auth_gateway_url": "http://localhost:8081",
  "l2": {
    "type": "sqlite",
    "filePath": "l2.db"
  }
}
```

> `auth_gateway_url` 為選填，若提供則寫入 L1，data-gateway 會自動向 auth-gateway 取得 JWT 公鑰。

安裝時自動建立以下**預設角色**（`使用者` collection，model type = `角色`）：

| ID | 名稱 |
|----|------|
| `使用者:角色:超級管理員` | `{"zh-tw":"超級管理員","en":"superadmin"}` |
| `使用者:角色:管理員` | `{"zh-tw":"管理員","en":"admin"}` |
| `使用者:角色:會員` | `{"zh-tw":"會員","en":"member"}` |
| `使用者:角色:貴賓` | `{"zh-tw":"貴賓","en":"vip"}` |
| `使用者:角色:黑名單` | `{"zh-tw":"黑名單","en":"blacklist"}` |

管理員記錄（model type = `使用者`）：

```json
{
  "id": "使用者:使用者:admin",
  "帳號": "admin",
  "密碼雜湊": "...",
  "角色": ["使用者:角色:超級管理員"],
  "updatedAt": "2026-07-22T..."
}
```

> 角色名稱使用 `@dui/smartmultilingual` 的 MultilingualString 格式，支援多國語言。

### 5.2 資料 API

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | `/api/:collection` | 列出指定 collection 下的所有 model type |
| GET | `/api/:collection/:model` | 列出指定 model type 的紀錄列表 |
| POST | `/api/:collection/:model` | 建立新紀錄 |
| GET | `/api/:id` | 以 composite ID 取得單筆紀錄 |
| PUT | `/api/:id` | 以 composite ID 更新單筆紀錄 |
| DELETE | `/api/:id` | 以 composite ID 刪除單筆紀錄 |

**ID 格式**：`collection:model:nanoid`（如 `圖片:標籤:abc123`）

> JWT 驗證統一由 `routes/api/_middleware.ts` 處理，接受任何有效 JWT（anonymous 或 authenticated）。
> Handler 可透過 `c.get('jwt_type')` 判斷目前是否已登入，彈性決定回傳資料範圍。

### 5.4 內部 API（供 auth-gateway 調用，不受安裝檢查限制）

| 方法 | 路徑 | 說明 |
|------|------|------|
| POST | `/inner-api/auth/verify-user` | 驗證使用者帳號密碼（回傳使用者資料） |

請求主體：

```json
{
  "帳號": "admin",
  "密碼": "password"
}
```

成功回傳：

```json
{
  "success": true,
  "data": {
    "id": "使用者:使用者:admin",
    "帳號": "admin",
    "角色": ["使用者:角色:超級管理員"]
  }
}
```

### 5.5 其他 API

| 方法 | 路徑 | 說明 |
|------|------|------|
| GET | `/health` | 健康檢查（回傳 L1/L2 狀態） |
| GET | `/logout` | 登出（清除 JWT Cookie，重新導向到 auth-gateway 登入頁） |

### 5.6 檔案路由

`routes/` 目錄下的頁面路由由 `@dui/framework` 的 `loadRoutes()` 自動載入：

| 路徑 | 檔案 | 說明 |
|------|------|------|
| `/` | `routes/index.tsx` | 首頁 |
| `/setup` | `routes/setup/index.tsx` | 安裝設定頁面 |
| `/admin` | `routes/admin/index.tsx` | 管理後臺頁面 |

## 6. 安裝流程

data-gateway 採用**首次使用設定**模式，類似 WordPress 的安裝體驗：

1. **啟動時檢查**：若 L1 中無 `l2_connection`，視為未安裝
2. **Middleware 阻擋**：未安裝時，所有 `/api/*` 路由（除了 `/api/setup`）與頁面路由（除了 `/setup`）被阻擋
3. **安裝請求**：`POST /api/setup` 接收 L2 資料庫設定、auth-gateway URL、管理員資訊
4. **儲存 auth-gateway URL**：若提供 `auth_gateway_url`，寫入 L1 供後續 JWT 公鑰取得使用
5. **初始化 L2**：建立 L2 連線、初始化 `使用者` collection
6. **建立預設角色**：自動建立超級管理員、管理員、會員、貴賓、黑名單等 5 個角色記錄
7. **建立管理員**：建立管理員使用者記錄（ID: `使用者:使用者:{帳號}`），角色為 `["使用者:角色:超級管理員"]`
8. **取得 JWT 公鑰**：若已設定 auth-gateway URL，自動向 auth-gateway 的 `GET /api/jwt-public-key` 取得 Ed25519 公鑰
9. **完成**：標記已安裝，解除路由限制

## 7. 安全設計

- **L2 連線資訊加密儲存**：存入 L1 前使用 `@dui/util` 的 `encrypt()` 加密
- **JWT 雙層驗證**：Middleware 接受任何有效 JWT，handler 自行判斷：\n  - **匿名 JWT**（`type: "anonymous"`）：僅含 tenant 資訊，1 小時有效期，handler 僅回傳公開資料\n  - **已認證 JWT**（`type: "authenticated"`）：含 tenant 與使用者身份，24 小時有效期，handler 回傳完整資料
- **JWT 本地驗證**：data-gateway **不儲存任何金鑰**，啟動後自動向 auth-gateway 的 `GET /api/jwt-public-key` 端點取得 Ed25519 公鑰並快取在記憶體中，所有 token 使用 EdDSA 演算法本地驗證
- **金鑰輪換支援**：若驗證失敗，data-gateway 會自動清除快取並重新向 auth-gateway 取得最新公鑰，支援無縫金鑰輪換
- **Tenant 路由**：從 JWT payload 的 `tenant` 欄位提取 domain name，自動切換至對應的 L3 資料庫，無需額外傳遞 domain 參數
- **密碼雜湊**：管理員密碼使用 bcryptjs 雜湊儲存
- **內部 API 保護**：`/inner-api/*` 不受安裝檢查限制，但僅供 gateway 內部調用

## 10. 環境變數

| 變數 | 預設值 | 說明 |
|------|--------|------|
| `DATA_GATEWAY_PORT` | `8002` | HTTP 監聽埠 |

> auth-gateway URL **不透過環境變數設定**，改由安裝流程（`POST /api/setup`）寫入 L1 的 `auth_gateway_url` 欄位。如此可在不重啟服務的情況下更新 auth-gateway 位置。
