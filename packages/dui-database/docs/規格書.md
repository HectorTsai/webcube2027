# @dui/database 規格書

> 最後更新：2026-07-22

---

## 1. 套件概述

`@dui/database` 是 WebCube2027 的資料庫抽象層，提供統一的資料存取介面，支援多種資料庫後端（Adapter 模式）。所有 Gateway 皆透過此套件操作資料庫，不直接依賴特定資料庫驅動。

**核心設計原則：純 JSON 存取，無需註冊 Model。**

- 存入資料為純 JSON
- 取出資料回傳純 JSON
- 使用者透過 data-gateway 拿到 JSON 後，自行從 ID 判斷轉成 Model（選用）
- 節省記憶體開銷，無需為每個紀錄建立 Model 實例

### 相依關係

```
@dui/database
  ├── @dui/kv      ← L1Store（輕量 KV store，僅設定儲存，不含 heavy deps）
  └── @dui/util    ← 加解密（encrypt/decrypt）、日誌（info/error/debug/warn）
```

## 3. 架構

### 三層模型

| 層級 | 類型 | 實作套件 | 儲存內容 | 生命週期 |
|------|------|---------|---------|---------|
| **L1 (CONFIG)** | JSON 檔案 (`l1.json`) | `@dui/kv` | 系統設定、L2 連線資訊（加密儲存） | 隨部署環境 |
| **L2 (SYSTEM)** | 中央資料庫（支援 9 種） | `@dui/database` | 全網站共用資料（字典、圖示、佈景主題等） | 系統級永久儲存 |
| **L3 (TENANT)** | 多種後端 | `@dui/database` | 個別租戶網站的私有內容與設定 | 租戶級永久儲存 |

> L1 不再是完整資料庫，而是純 KV 設定儲存。L1Store 已獨立為 `@dui/kv` 套件，`@dui/database` 僅 re-export 以維持向後相容。

### 3.1 查詢路由

- **查詢（讀取）**：L3（指定 host）→ 其他 L3 → L2（SYSTEM）。**不再降級到 L1。**
- **寫入/刪除**：寫入目標層級（host → L3、無 host → L2）
- **查詢所有列表**：合併 L3 + L2 資料，同 id 去重（高層級優先）

### 3.2 Collection 階層

```
Collection (實際的 table/collection)
  ├── Model: 圖片     →  id = "圖片:圖片:xxxx"
  ├── Model: 標籤     →  id = "圖片:標籤:ssss"
  └── Model: 評論     →  id = "圖片:評論:cccc"
```

**Collection 是實際儲存的資料表**，每個 collection 可包含多種 model type，以 composite ID 的第二段區分。

### 3.3 資料庫 ID 規則

格式：`collection:model:nanoid`

- **collection** — 集合/表格名稱（實際存在的 table/collection）
- **model** — 資料型態名稱（區分同一 collection 下的不同類型）
- **nanoid** — 唯一識別碼（12 字元）

範例：`圖片:圖片:abc123def456`、`圖片:標籤:xyz789`、`使用者:管理員:aB3xK9mW2pQr`

### 3.4 連線資訊管理

- L2 連線資訊存於 L1 KV store（加密後寫入 `l1.json`）
- L3 連線資訊存於 L2（SurrealDB 網站資訊）
- 不需要 `.env` 配置資料庫連線

## 4. 設定儲存（L1 — Config Store）

L1 的生命週期**不再由 `@dui/database` 管理**。由呼叫方（通常是 `@dui/framework` 的 `createGateway()`）負責：

1. 建立 `L1Store` 實例（來自 `@dui/kv`）
2. 呼叫 `l1.init()` 初始化
3. 呼叫 `dataPool.setConfigStore(l1)` 將 L1 參考傳入 Pool

Pool 本身不負責 L1 的建立與初始化，僅透過 `setConfigStore()` 取得參考。

### 4.1 `setConfigStore(l1: L1Store)`

提供 L1 Store 參考給 Pool，需在 `initL2()` 之前呼叫。

```typescript
dataPool.setConfigStore(l1);  // 傳入已 init 的 L1Store
```

### 4.2 `config` getter

取得 L1Store 實例（若未呼叫 `setConfigStore()` 則為 `null`）。

```typescript
const l1 = dataPool.config;  // L1Store | null
await l1?.get('some_key');
```

### 4.3 金鑰管理

加密金鑰**不再由此套件管理**。金鑰生命週期由 `@dui/framework` 的 `createGateway()` 統一處理：
- 金鑰儲存在 L1 的 `_crypto_key` 欄位
- 啟動時自動產生（若不存在）
- 透過 `registerKey()` 註冊到 `@dui/util` 的 crypto 模組

## 5. 資料池（Data Pool）

### 5.1 架構概述

**資料池**（`PoolCore`）是全域單例，所有 L2/L3 資料操作必須透過此入口。
Pool 專注於 L2/L3 連線管理與查詢路由，不再參與 L1 生命週期。

### 5.2 核心方法

| 方法 | 說明 |
|------|------|
| `setConfigStore(l1)` | 提供 L1 Store 參考（需在 initL2 前呼叫） |
| `getById<T>(id, host?)` | 查詢單筆資料（從 composite ID 解析 collection，L3 → L2 降級） |
| `list<T>(collection, modelType?, options?, host?)` | 從最高可用層取回分頁列表（可選篩選 model type） |
| `listAll<T>(collection, modelType?, options?, host?)` | 合併所有層級列表資料（同 id 去重） |
| `queryByField<T>(collection, filter, modelType?, host?)` | 依欄位值查詢（可選篩選 model type） |
| `listModelTypes(collection, host?)` | 列出 collection 下所有 model type |
| `create<T>(collection, id, data, host?)` | 建立新紀錄 |
| `update<T>(collection, id, data, host?)` | 更新（upsert）紀錄 |
| `delete(id, host?)` | 刪除資料 |
| `patch<T>(collection, id, fields, host?)` | 部分更新指定欄位，不走 read-modify-write |
| `count(collection, modelType?, host?)` | 計算記錄數（可選篩選 model type） |
| `initialize(collection)` | 初始化 collection（table） |
| `config` getter | 取得 L1Store 實例（`null` 若未設定） |
| `System` getter | 直接取得 L2 DatabaseAdapter \| `null` |

> **所有方法回傳純 JSON**，不進行 Model 實例轉換。

### 5.3 生命週期

```
呼叫方（如 @dui/framework createGateway()）
  → new L1Store(`${dataDir}/l1.json`)
  → await l1.init()
  → dataPool.setConfigStore(l1)          // 傳入 L1

dataPool（此套件管理）
  → await dataPool.initL2()              // 從 L1 讀取連線資訊 → 連接 L2 資料庫
  → await dataPool.initL3(host)          // L3 懶載入，getLayerKeys() 內建自動偵測觸發
```

## 6. BaseModel（選用工具）

### 6.1 概述

`BaseModel` 是一個選用的基底類別，**非必須使用**。如果你需要在應用層建立 Model 實例，可以直接 `new 你的Model(data)`，不需要註冊到資料庫層。

### 6.2 內建欄位

| 欄位 | 型別 | 說明 |
|------|------|------|
| `id` | `string` | 複合 ID，格式 `collection:model:nanoid` |
| `tags` | `string[]` | 標籤 ID 集合 |
| `updatedAt` | `Date` | 最後修改時間戳 |
| `collection` | `string` (getter) | 複合 ID 的 collection 部分 |

方法：

- `toJSON()` — 序列化為純物件
- `init()` — 初始化 hook

### 6.3 使用範例

```typescript
// 從 data-gateway 取得 JSON
const res = await fetch('/api/圖片/圖片:圖片:abc123');
const { data } = await res.json();  // data = { id: '...', tags: [...], ... }

// 選用：轉成 Model 實例（無需註冊）
import 圖片 from './models/圖片.ts';
const image = new 圖片(data);
console.log(image.網址);
```

## 7. Adapter 介面

每個資料庫 Adapter 須實作 `DatabaseAdapter` 介面：

```typescript
interface QueryOptions {
  limit?: number;
  offset?: number;
}

interface FieldFilter {
  field: string;
  value: string;
}

interface DatabaseAdapter {
  readonly type: string;
  getById(id: string): Promise<Record<string, unknown> | null>;
  list(collection: string, modelType?: string, options?: QueryOptions): Promise<Record<string, unknown>[]>;
  queryByField(collection: string, filter: FieldFilter, modelType?: string): Promise<Record<string, unknown>[]>;
  create(collection: string, id: string, data: Record<string, unknown>): Promise<Record<string, unknown>>;
  update(collection: string, id: string, data: Record<string, unknown>): Promise<Record<string, unknown>>;
  delete(id: string): Promise<boolean>;
  patch(collection: string, id: string, fields: Record<string, unknown>): Promise<Record<string, unknown> | null>;
  count(collection: string, modelType?: string): Promise<number>;
  initialize(collection: string): Promise<void>;
  listModelTypes?(collection: string): Promise<string[]>;  // 選用
}
```

## 8. 回應格式

所有資料池操作回傳 `QueryResult<T>`：

```typescript
interface QueryResult<T> {
  data: T | null;       // 查詢結果（純 JSON，失敗為 null）
  source: "L2" | "L3";  // 資料來源層級（L1 不參與查詢）
  success: boolean;     // 操作是否成功
  error?: string;       // 失敗時的錯誤訊息
}
```

---

## 9. 錯誤處理

資料池層級的操作失敗時，`QueryResult.error` 為純文字錯誤訊息（如 `"Not found in any available layer"`、`"Record exists in L2 but delete targets L3"` 等）。

具體錯誤碼（如 `NOT_FOUND`、`DELETE_PROTECTED` 等）為上層 API Gateway 的職責，不在此套件範圍內。

### 9.1 Adapter 錯誤日誌

各 Adapter 實作應使用 `@dui/util` 的 `info` / `error` 記錄操作失敗日誌，不應靜默吞掉例外。

**Firestore adapter 已知限制：**

為避免 Firestore 複合索引缺失問題，`queryByField()` 在指定 `modelType` 時採用**兩階段查詢**：
1. 先以 `data.{field}` equality 查詢（不需要複合索引）
2. 在記憶體中過濾符合 `modelType` 的結果

這意味著掃描的資料量會比直接在 Firestore 端過濾大，但避免了在每個新的 filter + modelType 組合上建立索引。
