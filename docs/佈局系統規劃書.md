# WebCube2027 佈局系統規劃書

## 📋 專案概述

### 🎯 功能目標
建立一個模組化、可擴展的動態佈局系統，讓 WebCube2027 能夠：
- 支援多種頁面佈局風格
- 透過資料庫動態切換佈局
- 提供簡單的擴展機制
- 確保系統穩定性和效能

- **2026-03-12** - 實作與測試
- **2026-03-12** - 完成與文件更新

## 🏗️ 系統架構

### 📁 檔案結構
```
src/components/
├── 佈局.tsx          # 統一佈局入口元件
└── 佈局/             # 佈局元件目錄
    ├── 經典.tsx      # 經典佈局 (header + main + footer)
    ├── 簡約.tsx      # 簡約佈局 (只有內容區域)
    └── 居中.tsx      # 居中佈局 (內容置中顯示)

src/routes/api/
└── layouts.tsx       # 佈局清單 API

database/models/
└── 骨架.ts          # 骨架模型，包含佈局欄位
```

### 🔄 資料流程
1. **使用者請求** → `_layout.tsx`
2. **讀取設定** → 從資料庫取得 `骨架.佈局`
3. **選擇佈局** → `佈局.tsx` 根據風格選擇對應元件
4. **渲染頁面** → 應用選定的佈局

## 🎨 佈局設計

### 📊 可用佈局

#### 🎨 經典佈局 (Classic)
- **用途** - 一般網頁，需要完整導航結構
- **結構** - Header + Main + Footer
- **特色** - 完整的頁面結構，包含導航選單

#### 📱 簡約佈局 (Minimal)
- **用途** - 專注內容，最小干擾
- **結構** - 只有內容區域
- **特色** - 極簡設計，突出內容

#### 🎯 居中佈局 (Centered)
- **用途** - 登入頁面、單一焦點頁面
- **結構** - 內容置中顯示
- **特色** - 視覺焦點集中

### 🔧 技術實作

#### 佈局元件設計
```typescript
interface 佈局Props {
  風格: string
  children: unknown
}

// 重要：佈局元件必須是同步的，不能使用 async
// 這是 Hono 框架的限制，_layout.tsx 不支援 async 元件
export default function 佈局({ 風格, children }: 佈局Props) {
  const LayoutComponent = 佈局風格表[風格] || 經典佈局
  return <LayoutComponent>{children}</LayoutComponent>
}
```

#### 設計考量
- **同步渲染**: 佈局元件必須是同步函數
- **靜態匯入**: 所有佈局必須在編譯時確定
- **穩定性優先**: 放棄動態載入換取更好的穩定性

#### 風格對應表
```typescript
const 佈局風格表 = {
  經典: 經典佈局,
  簡約: 簡約佈局,
  居中: 居中佈局,
  classic: 經典佈局,
  minimal: 簡約佈局,
  centered: 居中佈局
}
```

## 📡 API 設計

### 🔍 取得佈局清單 API

**端點**: `GET /api/layouts`

**功能**: 動態掃描 `components/佈局/` 目錄，回傳可用佈局

**回應格式**:
```json
{
  "成功": true,
  "佈局": ["經典", "簡約", "居中"],
  "數量": 3,
  "說明": "可用的佈局風格"
}
```

**錯誤處理**:
- 目錄不存在時回傳預設佈局
- 檔案讀取失敗時回傳預設佈局
- 確保 API 總是有回應

## 🛡️ 技術挑戰與解決方案

### 🚨 挑戰 1: Hono _layout 不支援 async 元件 (關鍵限制)
**問題**: Hono 的 _layout.tsx 不支援 async 元件，這是一個框架限制
**發現**: 雖然頁面元件可以使用 async，但 layout 元件必須是同步的
**解決**: 使用同步元件，靜態匯入所有佈局，放棄動態 import 方案
**影響**: 
- 無法實現真正的動態載入
- 必須在編譯時確定所有佈局
- 但換來更好的穩定性與效能

**重要備注**: 這個限制是 Hono 框架的內部機制，不是程式錯誤。在設計類似系統時必須考慮這個限制。

### 🚨 挑戰 2: children 渲染問題
**問題**: `Cannot convert object to primitive value`
**原因**: 在 async 環境中嘗試直接渲染複雜的 React 元件物件
**解決**: 直接渲染 children，避免 JSON.stringify 或其他轉換

### 🚨 挑戰 3: 檔案系統權限
**問題**: Deno 需要明確的檔案系統權限
**解決**: 在 deno.json 中加入 `--allow-write` 權限

### 🚨 挑戰 4: Node.js 模組匯入
**問題**: Deno 需要 `node:` 前綴
**解決**: 使用 `node:path` 而不是 `path`
