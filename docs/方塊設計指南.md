# 🧱 方塊設計指南（AI 2026 Headless 完全體專用提示詞）

> **【重要】** 本文件是全站方塊（Cube）JSON 圖紙的最高指導原則。AI 在生成、修改任何方塊 JSON 時，必須嚴格遵守以下 Schema、架構分工與禁令，否則會導致前端渲染崩潰或 DOM 結構污染。

---

## 1. 核心哲學：單一節點與數據驅動

本系統的 `方塊.tsx` 是一個 Headless 晶片底座。它追求「零垃圾 Wrapper 節點」。
一個 JSON 定義在遞迴解析到最底層時，只會渲染出一個對應的原生 HTML 標籤。所有的排版、五態、顏色，全部透過屬性供電與 UnoCSS 類名驅動。

## 2. 統一方塊 JSON Schema

```typescript
{
  // ── 1. 定義與繼承 ──
  from: string;                         // 來源指針。可以是原生標籤 ("div", "ul", "a") 或 DB 方塊 ID ("方塊:方塊:容器")
  tag?: string;                          // ⭐ 標籤覆寫（肉身）。最終留在 HTML 上的標籤（例如 <footer>、<section>）

  // ── 2. 樣式與供電 ──
  className?: string;                   // 基礎 CSS 類（限 UnoCSS 原子類）。不可寫死會變動的樣式！
  style?: Record<string, string>;       // 靜態行內樣式
  styleConditions?: Record<string, string>; // 條件 CSS。Key 必須對應一個 Boolean 類型的引數，& 會被自動替換為作用域類名

  // ── 3. 參數對外開放 (引數) ──
  args?: Record<string, {
    type: "string" | "number" | "boolean";
    description?: string;
    default?: unknown;
    options?: string[];
    variants?: Record<string, {          // 變體選項切換
      className?: string;                // 激活此選項時，追加入外層的類名
      style?: Record<string, string>;
      alpine?: object;
      on?: Record<string, string>;
      data?: Record<string, string>;
      [containerProp: string]: unknown;  // ⭐ 任意 key 會合併到 mergedArgs，傳遞給下游（如 rounded, padding）
    }>;
  }>;

  // ── 4. 參數內部鎖定 (預設值) ──
  defaults?: Record<string, unknown>;    // ⚠️ 內部鎖定的參數（如佈局、圓角）。不可與 args 的 key 重疊！

  // ── 5. 內容與插槽 ──
  children?: (object | string)[];        // 子方塊定義陣列，或純文字字串
  prepend?: PrependAppendItem[];         // 條件式前置內容（含 if 判定）
  append?: PrependAppendItem[];          // 條件式後置內容（含 if 判定）
  slots?: Record<string, {               // 具名插槽
    from: string;
    shareChildren?: boolean;
    children?: (object | string)[];
  }>;

  // ── 6. 結構包裹與互動 ──
  wrap?: { from: string; className?: string; style?: Record<string, string> }; // 群組穿衣包裹機制
  alpine?: { init?: string; bind?: Record<string, string>; attrs?: Record<string, string>; model?: string };
  on?: Record<string, string>;           // { "click": "open = true" } -> x-on:click
  data?: Record<string, string>;         // { "id": "{item.id}" } -> data-id="..."
  repeat?: string;                       // 陣列展開語法，例如 "{items}"
  mergedArgs?: Record<string, string>;   // 共享變數注入，支援內部 @api 呼叫（詳見 §8）
}

interface PrependAppendItem {
  if?: string;                           // 條件 Key（當引數為 falsy 時跳過渲染）
  from: string;
  className?: string;
  children?: (object | string)[];
  [key: string]: unknown;
}
```

## 3. AI 必知之四大核心機制與連動

### Mechanism 3.1：標籤覆寫（from 靈魂 vs tag 肉身）

- **from 決定靈魂（屬性繼承）**：決定這個方塊要載入哪張圖紙、繼承哪些變體（variants）與 defaults。
- **tag 決定肉身（最終 HTML 標籤）**：當你想讓一個方塊完全繼承 `"方塊:方塊:容器"` 的所有排版、圓角與陰影超能力，但最終在網頁上必須輸出為 `<footer>` 標籤以符合 SEO 語義時，必須使用 `tag` 進行覆寫。

### Mechanism 3.2：智慧五態供電（active、hover、selected、focus 與 activeStateName）

底層晶片內建了智慧五態判定，支援三種層級：

**層級 A：靜態 Boolean（純 CSS 狀態）**

當你傳入靜態 Boolean（如 `active: true`），它會渲染出 `data-active="true"`，由 UnoCSS 的 `[data-active]` 選擇器接管供電。

```json
{ "active": true, "hover": true }
```

**層級 B：Alpine 表達式（進階動態綁定）**

當你傳入運算式字串（含 `$` 或 `.`），它會智慧切換為 Alpine 動態綁定 `:data-active="$store.Menu.isOpen"`。

```json
{ "active": "$store.Menu.isOpen" }
```

**層級 C：activeStateName 宣告式動態切換（推薦，方塊內建機制）**

只需宣告 `activeStateName`，系統全自動生成三條 Alpine 指令：

| 自動生成 | 效果 |
|---|---|
| `x-bind:data-active` | 動態綁定 `data-active` 屬性，含 `?.` + `??` fallback 防呆 |
| `x-bind:style` | 斷電時自動注入 neutral 色 CSS 變數，覆蓋原色 |
| `x-init` | 防呆初始化 `Alpine.store('Container', {})` 及對應屬性 |

使用範例：

```json
{
  "args": {
    "activeStateName": {
      "type": "string",
      "description": "Alpine store 屬性名稱（用於動態切換）",
      "default": ""
    }
  }
}
```

呼叫端：

```tsx
// 簡單模式：僅傳屬性名，自動前綴 $store.Container
<Cube from="方塊:方塊:容器" activeStateName="btn_active_toggle" active hover />

// 完整路徑模式：直接指定 Alpine 表示式
<Cube from="方塊:方塊:容器" activeStateName="$store.Nav.isOpen" active hover />
```

對應的 Alpine 操控端：

```html
<button x-on:click="$store.Container.btn_active_toggle = !$store.Container.btn_active_toggle">
  點擊切換
</button>
```

> **AI 鐵律**：
> - `activeStateName` 與傳統的 `active: "$store.xxx"` 表達式**互斥**——前者生成 `x-bind:` 指令 + `x-init` + `x-bind:style`，後者僅生成 `:data-active`。
> - `activeStateName` 的路徑自動判別：不含 `$` 或 `.` 時自動前綴 `$store.Container.`；含 `$` 或 `.` 時直接使用。
> - 這兩個屬性會與佈景主題系統的「風格（Style）」模組聯動（利用 `[data-active]` 選擇器供電），嚴禁在 `className` 裡手動拼裝狀態類名（如 `active ? 'bg-blue-500' : 'bg-gray-200'`）。

### Mechanism 3.3：全自動顏色供電（color）

本系統擁有全自動的 UnoCSS 顏色映射。

> **AI 鐵律**：只要在引數中傳入 `color: "primary"`（支援 `primary`, `secondary`, `accent`, `neutral`, `info`, `success`, `warning`, `error`, `ghost`），底層會自動注入 `cube-color-{color}` 類名，並自動啟用佈景主題的 OKLCH 變數。嚴禁自創 `text-[#ff0000]` 或手動拼寫 `--c-current` 行內樣式。

### Mechanism 3.4：群組包裹穿衣（wrap）

當你需要製作一個列表、下拉選單或表格時，使用 `wrap` 屬性。底層晶片會自動進行語義推導：

- 當外層 `wrap.from` 為 `"ul"` 或 `"ol"` 時，內層的子項（children）會被自動包裹上 `<li>` 標籤，並套用斑馬紋電路。
- 當外層 `wrap.from` 為 `"select"` 時，內層子項會被自動包裹上 `<option>` 標籤。
- 其餘情況預設包裹 `<div>`。

> **AI 鐵律**：此時 `prepend` 與 `append` 可以完美繞過衣服規範，直接塞入如「列表標題」等非 `li` 結構。

## 4. 內部鎖定（defaults）vs 對外開放（args）

AI 設計 JSON 時，必須嚴格區分這兩者的邊界：

- **args**：使用者在後台或呼叫端可以傳入、切換的參數（如 `color`, `size`, `active`）。
  - args 的「預設值」寫在 arg 定義內部的 `default` 欄位。
- **defaults**：此元件傳給 `from` 繼承物件的**內定值**，強制鎖定、不容許外部覆寫（如頁尾永遠 `border: "none"`、永遠 `align: "center"`）。
  - `defaults` **不是「預設值」**，是寫死的內部固定值。兩者概念完全不同。

> ⚠️ **致命規則**：只要 key 進了 `defaults`，該 key **絕對不能再出現於 `args`**。因為 `defaults` 意味著「不給使用者選擇」，那就不該在 `args` 中暴露。

```json
// ✅ 正確：頁尾不讓使用者選 border / shadow / align / justify
{
  "from": "方塊:方塊:容器",
  "defaults": {
    "border": "none",
    "shadow": "none",
    "align": "center",
    "justify": "center"
  },
  "args": {
    "color": { "type": "string", "default": "base" },
    "padding": { "type": "string", "default": "xs" }
    // ← 沒有 border / shadow / align / justify，因為已被 defaults 鎖定
  }
}
```

## 5. 變體（Variant）設計的絕對原則

當容器的幾何特徵（如圓角 `rounded`、內距 `padding`）需要跟隨某個引數（如 `size`）變化時，必須透過變體屬性（Variant Prop）直接傳遞，絕對不要拼寫在 `className` 字串中。

### ❌ 錯誤示範

AI 亂編的 className，會與佈景主題幾何 Token 打架：

```json
{
  "variants": {
    "sm": { "className": "rounded-sm p-2 w-32" },
    "lg": { "className": "rounded-lg p-6 w-48" }
  }
}
```

### ✅ 正確示範

利用變體屬性穿透，完美與佈景主題系統無縫接軌：

```json
{
  "args": {
    "size": {
      "type": "string",
      "default": "md",
      "options": ["sm", "md", "lg"],
      "variants": {
        "sm": { "rounded": "sm", "padding": "xs", "width": "32" },
        "md": { "rounded": "md", "padding": "sm", "width": "40" },
        "lg": { "rounded": "lg", "padding": "md", "width": "48" }
      }
    }
  }
}
```

## 6. 常用禁令與黃金法則速查表

| ❌ 嚴禁行為 | ✅ 正確做法 |
|---|---|
| 方塊的 `from` 等於自己的 id | 一律引導至 `"from": "方塊:方塊:容器"`（僅容器/圖示/圖片除外） |
| 擅自拼寫 `text-[#ff0000]` 或 `bg-blue-500` | 使用全自動 `color` 參數，或使用全站 preset 原子類 |
| 在 `className` 裡用三元運算子寫死狀態樣式 | 傳遞 `active` / `hover` 引數，讓佈景主題的「風格」模組接管 |
| 在 `children` 中使用 `{ "$api": { "path": "..." } }` 動態抓取外部 API | 改用 `mergedArgs` 定義 `@api` 內部變數，children 中以 `{變數名}` 插值引用（詳見 §8） |
| 設計 `variant`（樣式變體）arg 來切換 solid/outline/ghost 等外觀 | 已廢棄。樣式變體已全面轉移至「風格（Style）」模組，由佈景主題統一管轄 |
| 呼叫複合方塊時忘記傳遞 `context` | 外層呼叫處一律綁定 `context={c}`，確保非同步 DB 載入鏈不中斷 |
| 在 `className` 中寫死 `bg-transparent` 覆蓋背景 | 背景由風格系統全權控制（active → `bg-current`，inactive → `bg-base-50`） |
| 同時使用 `cursor-not-allowed` 與 `pointer-events-none` | 兩者互斥：pointer-events-none 會讓游標穿透，無法顯示禁止符號 |

## 7. 斷電與 Disabled 狀態的系統級行為

當方塊的 `active={false}` 或 `disabled={true}` 時，`方塊.tsx` 會**全自動**執行以下防線，AI 不需手動處理：

| 自動行為 | 機制 | 適用範圍 |
|---------|------|---------|
| `cursor-pointer` → `!cursor-not-allowed` | 正則替換 className | 所有元素 |
| `aria-disabled="true"` | 注入無障礙標記 | 所有元素 |
| 移除 `href`、`action`、`formaction` | `原生點擊屬性` 清單 | 所有元素 |
| `data-active="false"` → inactive 風格 CSS | UnoCSS 規則 `c-style-apply` | 掛載 `c-style-apply` 的元素 |

> **AI 鐵律**：設計超連結類方塊時，不要寫死 `bg-transparent`（會蓋掉 inactive 灰底），不要自己加 `pointer-events-none`（會讓 `cursor-not-allowed` 失效）。這些全部由系統自動處理。

## 8. API 呼叫的雙軌機制：`$api`（禁止）vs `@api`（允許）

方塊系統提供兩種 API 呼叫方式，但用途與安全邊界截然不同：

### 8.1 `@api` 內部共享變數（`mergedArgs`）— ✅ 允許

在 `mergedArgs` 中使用 `@api/...` 規格字串，系統會在 `準備核心Args` 階段透過 `InnerAPI` 內部調用對應 API，並將結果注入 `mergedArgs`，供 `{變數名}` 插值使用。

**機制**：
```
mergedArgs → resolveMergedArgsDef() → InnerAPI(context, path) → data 注入 mergedArgs
```

**範例** — 頁尾方塊使用 `@api` 抓取年份、翻譯文字：

```json
{
  "from": "方塊:方塊:容器",
  "mergedArgs": {
    "currentYear": "@api/v1/temporal/fullYear",
    "prefix": "@api/v1/translate/copyright",
    "suffix": "@api/v1/translate/all rights reserved"
  },
  "children": [
    {
      "from": "span",
      "text": "{prefix} \u00a9 {copyright.開始年份}-{currentYear}"
    }
  ]
}
```

**解析流程**：
1. `resolveMergedArgsDef` 將 `@api/` 替換為 `/api/`，呼叫 `InnerAPI(context, "/api/v1/temporal/fullYear")`
2. 取回 `{ success: true, data: 2026 }`，將 `currentYear = 2026` 寫入 `mergedArgs`
3. `children` 中的 `{currentYear}` 由 `substitute()` 自動替換為 `2026`

> **鐵律**：`@api` 只能用在 `mergedArgs` 中，不可出現在 `children`、`slots` 或其他欄位。

### 8.2 `$api` 外部動態引用（`children` / `slots`）— ❌ 禁止

在 `children` 或 `slots` 中使用 `{ "$api": { "path": "...", "field": "..." } }` 語法，會在渲染階段動態呼叫外部 API。

**禁用原因**：
- 違反數據驅動原則：API 資料應在 `mergedArgs` 階段統一解析，不應散落在 children 定義中
- 破壞方塊的可預測性：外部 API 的副作用無法被 `defaults` 鎖定或 `args` 覆蓋
- 增加渲染鏈複雜度：每個 child 獨自發起 API 呼叫，難以除錯與效能優化

**正確做法**：將需要的資料定義在 `mergedArgs` 中，透過 `@api` 統一解析，children 中以 `{變數名}` 引用。
