# 開發流程書

請閱讀 `docs/規格書.md`，並按照以下階段執行開發任務。全程遵守規格書定義的命名規則（§3.I）、API 回應格式（§3.F）、日誌規範（§3.K）及**統一資料庫訪問架構（§3.B）**。

## 🚨 重要架構限制

**嚴格禁止直接資料庫訪問**：
- ❌ 禁止直接使用 `取得KV資料庫()`
- ❌ 禁止直接建立 SurrealDB 連線
- ❌ 禁止直接操作 L1/L2/L3 資料庫

**🔧 分層權限限制**：

| 層級 | ✅ 允許操作 | ❌ 禁止操作 |
|------|------------|------------|
| **Middleware** | Context 操作、三層查詢管理器、直接資料庫 | 無 |
| **API 層** | Context 取得、三層查詢管理器 | 直接資料庫連線 |
| **Service 層** | Context 取得（預載資訊） | 三層查詢管理器、直接資料庫 |
| **Renderer 層** | API 調用 (`app.request`) | Context、Service、直接資料庫 |

**關鍵規則**：
- ✅ **只有 Middleware 和 API 層**可以使用 `三層查詢管理器`
- ✅ **只有 Middleware 層**可以直接操作資料庫連線
- ✅ **Service 層**只能使用 Context 中的預載資訊
- ✅ **Renderer 層**只能透過 API 調用獲取資料

**AI 編輯器注意**：必須遵循此分層架構限制！

# 第一階段 - 基礎設施與開發規範

對應規格書：§2 技術棧、§3.F API 回應格式、§3.I 命名規則、§3.K 日誌系統、§4 目錄結構

### 1. 專案初始化
- 建立 `deno.json`，配置 `imports`（包含 `hono`, `@unocss/preset-wind4`, SurrealDB client, `@dui/smartmultilingual` 等）。
- 建立 `.env` 檔案，配置 6 個環境變數：`SECRET_KEY`, `CACHE_ENABLED`, `AI_API_KEY`, `AI_PROVIDER`, `LOG_LEVEL`, `LOG_TO_FILE`。
- 建立專案目錄結構（依規格書 §4）：`/core`, `/database/models`, `/database/seeds`, `/middleware`, `/services/apiService`, `/services/mediaService`, `/services/rendererService`, `/components`, `/utils`。

### 2. 日誌系統
- 建立 `utils/logger.ts` 統一日誌介面（依規格書 §3.K）。
- 支援 `LOG_LEVEL` 等級控制（debug / info / warn / error）及 `LOG_TO_FILE` 檔案輸出。
- 全專案禁止直接使用 `console.log`，統一透過 Logger。

### 3. API 回應格式標準
- 建立統一的回應工具函式（依規格書 §3.F）：
    - 成功回應：`{ 成功: true, 資料, 來源 }`
    - 錯誤回應：`{ 成功: false, 錯誤: { 代碼, 訊息, 狀態碼 } }`
- 定義錯誤 Code 常數：`NOT_FOUND`, `FORBIDDEN`, `DELETE_PROTECTED`, `VALIDATION_ERROR`, `BAD_REQUEST`, `INTERNAL_ERROR`。

### 4. L1 (Deno KV) 核心
- 在 `/core/kv.ts` 實作 Deno KV 連線管理。
- 建立系統資訊 Model（含網站名稱、描述、商標等 SEO 欄位，依規格書 §3.G）。
- 建立 Seed 資料機制，初始化系統資訊預設值。

### 5. 內部調用驗證 (app.request)
- 在 `main.ts` 初始化 Hono App。
- 建立測試路由 `GET /api/v1/system/info`，從 Deno KV 讀取系統資訊。
- 實作內部調用測試：透過 `app.request` 驗證 API-First 架構運作正常。

### 驗收條件
- 目錄結構完整建立。
- Logger 可正常輸出各等級日誌。
- API 回應統一格式。
- KV 讀寫正常，內部調用流程正常。

# 第二階段 - 三層資料庫與核心 API

對應規格書：§2 Databases、§3.A 資料存取層、§3.C 多國語言

### 1. 多租戶解析中間件 (Tenant Resolver)
- 建立 `middleware/db-resolver.ts`：
    - 根據 Request 的 `host` 或 header，從 L1 (KV) 取得 L2 連線資訊，再從 L2 取得 L3 連線資訊。
    - 實作連線池 (Connection Pool)，將 DB 實例掛載至 Hono Context。
    - 連線失敗時優雅降級（L3 不存在則僅用 L2，L2 不存在則僅用 L1）。

### 2. Model / Seed 系統
- 建立核心 Model（依規格書 §3.A，繁中命名）：
    - `骨架`、`配色`、`佈景主題`、`頁面`、`方塊`。
    - 所有 Model 包含共通欄位 `可刪除: boolean`。
    - 多國語言欄位使用 `MultilingualString` / `SmartContent`（依規格書 §3.C）。
- 建立對應的 Seed JSON 檔案，Seed 資料 `可刪除 = false`。
- Middleware 連線 DB 後自動檢查並初始化 Seed 資料。

### 3. 核心 CRUD API（更新為 KV seeds 邏輯）
- **預設值 API**（新增）：
    - `GET /api/v1/defaults` - 取得所有預設值（theme/color/skeleton）
    - `GET /api/v1/defaults/theme` - 取得預設佈景主題
    - `GET /api/v1/defaults/color` - 取得預設配色
    - `GET /api/v1/defaults/skeleton` - 取得預設骨架
    - 預設值來源：L3 → L2 → L1 (KV seeds)，保證有回傳值

- **完整 CRUD API**：
    - `api/v1/themes`、`api/v1/themes/:id`、`api/v1/theme`（網站主題）
    - `api/v1/colors`、`api/v1/colors/:id`
    - `api/v1/skeletons`、`api/v1/skeletons/:id`
    - `api/v1/pages`、`api/v1/pages/:id`、`api/v1/pages/path/:path`（含巢狀路徑）
    - `api/v1/cubes`、`api/v1/cubes/:id`
- **三層查詢架構**：L3 → L2 → L1 (KV seeds)，保證預設值存在
- **系統 Seeds 保護**：
    - DELETE 檢查 `可刪除` 欄位，系統 seeds (`可刪除: false` 且來自 L1) 無法刪除
    - 回傳 `SYSTEM_SEED_PROTECTED` 錯誤碼
- **API 回傳格式**：包含 `success`, `data`, `source`, `error` 欄位
- **查詢參數**：支援 `?include_seeds=false` 過濾系統 seeds

### 4. 樣式資源 API
- `GET /api/v1/styles/available`：取得可用樣式資源。
- `GET /api/v1/styles/classes`：取得自訂 preset 的可用 class 列表（供 AI 生成用，依規格書 §3.J）。

### 驗收條件
- 所有 CRUD API 正常運作，回應格式統一。
- 三層查詢 fallback 機制正確。
- Seed 資料自動初始化。
- DELETE 保護機制有效。

# 第三階段 - 渲染引擎、UnoCSS 與 Media Service

對應規格書：§2 Styling、§3.B 渲染管線、§3.E Media Service、§3.G SEO、§3.H 快取

### 1. UnoCSS Runtime 生成器
- 建立 `core/unocss.ts`：
    - 配置 UnoCSS Generator，引入 `@unocss/preset-wind4`。
    - 建立自訂 preset（學習 daisyUI 設計理念），定義顏色、字型、間距。
    - 使用 CSS Variables + oklch 色彩格式。
    - 實作 `產生樣式(html: string)` 函式。

### 2. 路由分發器
- 在 `main.ts` 實作 `app.get('*')` 動態分發器（依規格書 §3.B）：
    - `/api/*` → apiService
    - `/medias/*` → mediaService
    - 其餘 → rendererService
    - 各 Service 自行處理錯誤及 404。

### 3. Renderer Service
- 建立 `services/rendererService`：
    - 透過 `app.request` 向 API 取得頁面資料、骨架佈局、佈景主題、配色等。
    - 組件解析：預設方塊用 `import()`，AI 方塊用 JSX 字串解析器，組合方塊支援巢狀。
    - 渲染完成後呼叫 UnoCSS 產生 CSS，注入 `<head><style>`。
    - CSS Variables 掛載到 `<html>` style 屬性。
    - SEO meta tags 從網站資訊取得，fallback 到系統資訊（依規格書 §3.G）。
    - 404 時渲染錯誤頁面。

### 4. Media Service
- 建立 `services/mediaService`（依規格書 §3.E）：
    - 列表 API：`GET /medias/{type}` 支援分頁（page, pageSize）、排序（sort, order）、搜尋（search）。
    - 單筆 API：`GET /medias/{type}/:id` 回傳媒體本身。
    - 特殊路徑：`/medias/icon` 抓取網站/系統資訊的商標作為 favicon。
    - 各類型找不到時回傳對應的預設資源（非 JSON 錯誤）。
    - SVG 從 SurrealDB 讀取。

### 5. 快取機制
- 實作快取層，透過 `.env` 的 `CACHE_ENABLED` 控制開關（依規格書 §3.H）。
- 開發環境預設關閉。

### 驗收條件
- 完整的頁面可在瀏覽器渲染，包含 CSS 與 CSS Variables。
- Media Service 可回傳媒體或預設資源。
- favicon 路徑正常運作。
- SEO meta tags 正確注入。
- 快取開關可控制。

# 第四階段 - AI 方塊生成與安全管線

對應規格書：§3.J AI 方塊生成規範

### 1. Prompt Engine
- 建立 `utils/ai-prompts.ts`，定義標準 System Prompt（依規格書 §3.J.3）：
    - 僅使用 Hono JSX 語法。
    - 優先使用自訂 preset class（透過 `/api/v1/styles/classes` 動態取得，注入 Prompt）。
    - 可搭配 Tailwind 原生 utility class。
    - 禁止 arbitrary values（如 `text-[#ff0000]`）。
    - 禁止 `<script>` 標籤及 inline event handlers。
    - 組件必須接收 `props`。

### 2. 安全過濾器 (Sanitizer)
- 建立 `utils/sanitizer.ts`（依規格書 §3.J.4）：
    - 使用 `deno-dom` 解析 HTML/JSX 字串。
    - 移除所有 JavaScript 執行路徑（`<script>`、`javascript:` URI、inline events）。
    - JSX 語法檢查器：解析失敗則拒絕寫入 DB。

### 3. AI-to-DB Pipeline
- 建立 `POST /api/v1/ai/generate-cube`（依規格書 §3.J.1）：
    - 請求 Body：`{ "描述": "...", "目標": "system" | "tenant" }`
    - 流程：取得可用 class → 注入 Prompt → LLM 生成 → 安全過濾 → 語法驗證 → 存入 DB。
    - 支援多 LLM Provider（透過 `.env` 的 `AI_PROVIDER` / `AI_API_KEY`）。

### 4. JSX 字串渲染安全
- 渲染前二次安全過濾（防止 DB 被篡改）。
- 解析失敗時顯示錯誤佔位元件，不影響其他方塊。

### 5. 失敗處理
- LLM 呼叫失敗：`500 INTERNAL_ERROR`。
- 產出不合法：`422 VALIDATION_ERROR`，可自動重試一次。
- 渲染失敗：佔位元件 + 日誌記錄。

### 驗收條件
- AI 可成功生成方塊並存入 DB。
- 安全過濾器可攔截惡意內容。
- 生成的方塊可在頁面上正確渲染，UnoCSS 產生對應樣式。
- 失敗場景不會導致伺服器崩潰。

# 第五階段 - 認證授權與整合優化

對應規格書：§3.D 認證/授權系統

### 1. 使用者與角色 Model
- 建立「使用者」和「角色」兩個 Model。
- 實作認證機制（JWT 或 Session，待定）。
- 密碼加密使用 `.env` 的 `SECRET_KEY`。

### 2. API 權限保護
- POST / PUT / DELETE 需認證後才能存取。
- GET 保持開放（或依需求設定）。
- 根據角色區分系統管理員 / 網站管理員權限。

### 3. 整合測試
- 端到端流程測試：
    - 系統管理員設定 L2 連線 → 網站管理員申請網站 → 設定佈景主題 → AI 生成方塊 → 頁面渲染。
- 效能測試：確認快取開啟後的回應時間改善。
- 安全測試：確認未認證的寫入請求被拒絕。

### 4. 生產環境準備
- `LOG_TO_FILE=true` 測試。
- `CACHE_ENABLED=true` 測試。
- 錯誤處理全面檢查（各 Service 的 404 / 500 處理）。

### 驗收條件
- 認證流程完整，權限控制有效。
- 完整端到端流程可執行。
- 生產環境配置就緒。
