# WebCube2027 架構概覽

> 最後更新：2026-07-24（多租戶架構、JWT 雙層設計、Gateway 設定流程）

---

## 專案定位

WebCube2027 是一個多閘道（Multi-Gateway）架構的 Web 應用平台，採用 Deno 2 + Hono 技術棧。核心設計理念是將不同的業務領域拆分為獨立的 Gateway 服務，共享底層的 Packages 層。

---

## 核心架構決策

### 多租戶識別：Domain = Tenant ID

每個租戶以其 **domain name** 作為唯一識別碼，不另建對照表。L2 中的 `網站資訊` collection 以 domain 為 ID，記錄租戶的 L3 資料庫連線設定與網站設定。

```
www.dui.com.tw  → tenant = "www.dui.com.tw"
dui.com.tw      → tenant = "dui.com.tw"（與上方視為不同網站）
```

### JWT 雙層設計

所有請求都必須攜帶 JWT（由 auth-gateway 簽發）。JWT 分兩種層級：

| | 匿名 JWT | 已認證 JWT |
|---|---|---|
| **簽發時機** | 首次連線（無 JWT 時） | 使用者登入成功後 |
| **內容** | `{ tenant, type: "anonymous" }` | `{ tenant, 帳號, 角色, type: "authenticated" }` |
| **有效期限** | 1 小時 | 24 小時 |
| **權限** | 讀取公開資料 | 完整 CRUD |

### JWT Context 注入規則

每個 Gateway 在驗證 JWT 通過後，應將 payload 中的關鍵資訊注入 Hono 的 `c.set()` context，供後續 handler 使用。注入欄位如下：

| `c.get(...)` | 型態 | 來源 | 說明 |
|---|---|---|---|
| `jwt_payload` | `object` | payload 全部 | 完整 JWT payload，包含所有原始欄位 |
| `tenant` | `string` | `payload.tenant` | 租戶 domain（如 `example.com`） |
| `jwt_type` | `string` | `payload.type` | `"anonymous"`（匿名）或 `"authenticated"`（已認證） |

Handler 可透過 `c.get('jwt_type')` 判斷請求來自匿名訪客或已登入使用者，彈性決定回傳資料範圍：

- `"anonymous"` — 僅回傳公開資料
- `"authenticated"` — 回傳完整資料（含登入者專屬內容）

此規則適用於所有 Gateway，各 Gateway 的 `_middleware.ts` 應統一實作此注入邏輯。

### 租戶資料流

```
請求進入 → 攜帶 JWT → data-gateway
  → 驗證 JWT（Ed25519 公鑰，本地驗證）
  → 從 JWT 取出 tenant
  → 查 L2 網站資訊取得 L3 連線設定
  → 切換至對應的 L3 資料庫
  → 執行查詢
```

## 三層儲存

| 層級 | 名稱 | 用途 | 管理方式 |
|------|------|------|----------|
| **L1** | 本地設定 | KV Store（JSON 檔案），存放加密金鑰、L2 連線資訊、各 gateway 的互連設定 | 各 gateway 自行管理 |
| **L2** | 系統資料庫 | 所有租戶共享，存放使用者帳號、角色、網站資訊（含 L3 連線設定） | data-gateway 管理 |
| **L3** | 租戶資料庫 | 每租戶獨立，存放該租戶的頁面內容、方塊、多媒體等業務資料 | data-gateway 管理 |

## 整體架構

```
packages/          ← 共用套件層（跨所有 gateway）
gateway/           ← 業務閘道層（獨立 HTTP 服務）
main.ts            ← 根入口（開發用單一啟動點）
deno.json          ← workspace 根設定 + 共用 imports
```

> **📖 規格書優先**：每個 Package 與 Gateway 的根目錄皆包含 `docs/規格書.md`，詳細說明該模組的架構設計、API 定義與使用方式。若有任何疑問，請先查閱對應的規格書。

### 資料流

```
[Browser]
    │
    ▼
[WebCube] ──(JWT cookie)──→ [auth-gateway] ──簽發/驗證 JWT
    │                              │
    │         ┌────────────────────┘
    │         ▼
    │   [data-gateway] ──@dui/database──→ [L2 (System DB)]
    │         │                              │
    │         │                              └── 網站資訊 → L3 連線設定
    │         └──────────────────────────────────→ [L3 (Tenant DB)]
    │
    ├──→ [page-gateway]  ──→ [data-gateway]（帶 JWT）
    ├──→ [render-gateway] ──→ [data-gateway]（帶 JWT）
    └──→ [其他 gateway]    ──→ [data-gateway]（帶 JWT）
```

### JWT 取得流程

任何服務發現請求無 JWT 時，向 auth-gateway 請求匿名 JWT：

```
[WebCube / 任一 Gateway]
  ├── 有 JWT cookie？→ 驗證 → 使用
  └── 無 JWT？→ POST /api/anonymous-token（帶 domain）
                → auth-gateway 簽發匿名 JWT { tenant, type: "anonymous" }
                → 設定為 cookie
                → 繼續處理請求
```

| **dui-multilanguage** | `@dui/smartmultilingual` | 0.3.0 | 多國語言資料型態（MultilingualString、SmartContent），含翻譯服務介面（預設 Google Translate），可注入自訂翻譯實作 |
| **dui-translation** | `@dui/translation` | 0.1.0 | **籌備中** — AI 驅動的翻譯服務（AI → data-gateway 快取 → Google fallback），待 ai-gateway 完成後實作 |
| **dui-util** | `@dui/util` | 0.1.0 | 通用工具（InnerAPI 內部 API 代理、Logger 日誌記錄） |

### Package 設計原則

- **重複性高的基礎設施** → package（如資料庫、多國語言）
- **特定 gateway 的工具** → 留在自己的 `utils/` 目錄
- **Package 之間的依賴**透過 workspace 本地解析，不經過遠端 JSR

## Gateway（業務閘道層）

| Gateway | 狀態 | 職責 |
|---------|------|------|
| **auth-gateway** | ✅ 基礎閘道 | 認證中心：JWT 簽發（匿名/已認證）、金鑰對管理、統一驗證 |
| **data-gateway** | ✅ 基礎閘道 | 資料中心：RESTful CRUD API、L1/L2/L3 管理、JWT 本地驗證、tenant-aware 資料庫切換 |
| **WebCube** | 🔧 規劃中 | 網站入口：UnoCSS + Alpine.js、資料庫路由、協調各 gateway 產出 HTML |
| **render-gateway** | 🔧 既有 | 方塊渲染引擎（UnoCSS），尚未拆分 |
| **page-gateway** | 📦 待拆分 | 網頁中心，處理 AI 產生的網頁 |
| **cube-gateway** | 📦 待拆分 | 元件中心，處理 AI 產生的方塊 |
| **theme-gateway** | 📦 待拆分 | 佈景主題中心 |
| **ai-gateway** | 🔧 既有骨架 | AI 池，管理 LLM 調度，提供 OpenAI 相容 API |
| **media-gateway** | ❌ 規劃中 | 圖片/影片處理中心 |
| **排程閘道** | ❌ 規劃中 | 工作排程中心 |

### Gateway 設計原則

- 每個 Gateway 是獨立的 HTTP 服務
- 所有 Gateway 共用 auth-gateway 簽發的 JWT 進行身份識別
- 每個 Gateway 透過自己的 **Setup 流程**（首次執行時）設定相依的 gateway URL
- 內部服務間通訊攜帶原有 JWT，確保 tenant 與身份正確傳遞
- 管理型 Gateway（ai、page、cube、theme、media）共用 `@dui/framework`（hono + tailwindcss + daisyUI + fileRouter）
- render-gateway 例外，使用 UnoCSS 渲染引擎
- WebCube 例外，使用 UnoCSS + Alpine.js + 資料庫路由

> **URL 不可硬編碼原則**：雖然各 Gateway 的 `main.ts` 提供預設埠號（如 auth-gateway 預設 8001、data-gateway 預設 8002），但部署時安裝者可將各服務安裝在不同主機或不同埠號。因此，**任何程式碼不得直接硬編寫相依服務的 URL 或埠號**。所有跨 Gateway 的 URL 必須從 L1 設定（安裝時寫入）或環境變數取得。此原則適用於中介層、HTML 頁面中的連結、重新導向目標、以及後端服務客戶端（如 dataGwClient）等所有場景。

## Gateway 啟動與設定流程

所有 Gateway 採用**安裝時設定**模式（類似 WordPress）。每個 Gateway 首次啟動時自動進入 `/setup`，由安裝者填寫相依服務的位置。

> 各服務的埠號預設值定義於 `main.ts`，但安裝者可自由指定。設定流程中所有跨服務 URL 均寫入 L1，執行期不得有任何硬編碼。

### 設定順序

```
1. auth-gateway                        ← 先起，因為所有 gateway 都需要它簽發 JWT
   └── Setup 填寫：data-gateway URL     ← 寫入 L1
   └── 初始化 Ed25519 金鑰對             ← 寫入 L1

2. data-gateway                        ← 第二個起，提供資料存取
   └── Setup 填寫：auth-gateway URL、L2 資料庫設定、管理員帳號
   └── 初始化 L2、建立預設角色與管理員
   └── 從 auth-gateway 取得公鑰，啟用 JWT 驗證

3. 其他 Gateway（WebCube、page、render 等）
   └── Setup 填寫：auth-gateway URL、data-gateway URL
   └── 設定完成後即可透過 JWT 存取資料
```

> auth-gateway 即使尚未設定 data-gateway URL，仍可獨立簽發**匿名 JWT**（僅含 tenant 資訊，不含使用者身份）。登入功能則需 data-gateway URL 設定完成後才生效。

## 主目錄 `main.ts`

根目錄的 `main.ts` 是開發階段的統一啟動點，負責：

1. 呼叫 `設定App()` 註冊 InnerAPI 實例
2. 啟動各 Gateway 服務
3. 設定全域中介軟體（CORS 等）

## 開發指令

| 指令 | 說明 |
|------|------|
| `deno task dev` | 啟動所有服務（開發模式） |
| `deno task dev:auth` | 啟動 auth-gateway |
| `deno task dev:data` | 啟動 data-gateway |
| `deno task dev:ai` | 啟動 ai-gateway |
| `deno task dev:render` | 啟動 render-gateway |
| `deno task start` | 正式啟動 |
