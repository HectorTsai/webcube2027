# WebCube2027 檔案路由器

## 概述
WebCube2027 檔案路由器是一個基於檔案系統的高效能自動路由系統，採用 **Hono 原生 JSX 渲染機制**，支援非同步組件和嵌套佈局。

## 🚀 核心功能

### 1. 自動路由系統
- **自動掃描** `src/routes` 目錄下的 `.tsx` 檔案
- **路由映射**：檔案路徑 → URL 路徑
- **動態路由**：支援參數化路徑（如 `/users/[id].tsx`）

### 2. 特殊檔案支援
- **`_layout.tsx`** - 嵌套佈局包裝器
- **`_middleware.tsx`** - 嵌套中間件
- **`_app.tsx`** - 全域 App 包裝器（最外層 HTML 結構）
- **`_404.tsx`** - 404 錯誤頁面
- **`_error.tsx`** - 錯誤處理器
- **`_500.tsx`** - 500 伺服器錯誤頁面

### 3. Hono 原生 JSX 渲染
- **零序列化器**：使用 Hono 內建的 JSX 渲染引擎
- **非同步組件支援**：完整支援 `async` 組件
- **串流渲染**：支援 Hono 的串流 JSX 渲染
- **類型安全**：完整的 TypeScript 支援

async wrapComponent(handler: Function, ctx: Context, path: string): Promise<unknown> {
  let element = await handler(ctx)
  
  const chain = this.getSpecialFilesChain(path).reverse()
  
  for (const specialFiles of chain) {
    if (specialFiles.layout) {

      const content = element  // 建立穩定快照

  }

## �🛡️ 內部重導向功能

### 設計理念
- **開發者主導**：提供兩種重導向方法，讓開發者根據情境選擇
- **高效能優先**：預設使用內部重導向，避免重新 request
- **安全閥機制**：循環時自動降級為外部重導向

### API 設計
```typescript
// Context 擴展
declare module 'hono' {
  interface Context {
    /** 內部重導向（高效能，避免重新 request） */
    internalRedirect(path: string): Promise<Response>
    /** 重導向鏈追蹤（防止循環） */
    _redirectChain?: string[]
  }
}
```

### 使用範例
```typescript
// _middleware.tsx 或路由組件中
export default async function AuthMiddleware(ctx: Context, next: Next) {
  const user = await getCurrentUser(ctx)
  
  if (!user) {
    // 內部重導向：高效能，適合權限檢查
    return await ctx.internalRedirect('/login')
  }
  
  if (!user.emailVerified) {
    // 外部重導向：需要完整頁面重新載入
    return ctx.redirect('/verify-email')
  }
  
  await next()
}
```

### 自動降級機制
```typescript
// 當檢測到循環時，自動降級為外部重導向
if (ctx._redirectChain.includes(targetPath)) {
  console.warn('檢測到循環，降級為外部重導向')
  return ctx.redirect(targetPath)
}
```

### 效能優勢
- **99% 情況**：使用內部重導向，無新 request
- **避免重複執行**：middleware 鏈不會重新執行
- **減少資料庫查詢**：權限檢查等操作不會重複

### 安全特性
- **循環檢測**：自動檢測重導向循環
- **次數限制**：防止超過 5 次的重導向鏈
- **自動降級**：循環或超限時降級為標準 302
- **路由驗證**：確保目標路由存在
- **除錯日誌**：記錄重導向過程

### 重導向保護機制
```typescript
// 完整的保護邏輯
ctx.internalRedirect = async (targetPath: string): Promise<Response> => {
  // 1. 檢查循環重導向
  if (ctx._redirectChain!.includes(targetPath)) {
    console.warn(`檢測到循環: ${ctx._redirectChain!.join(' -> ')} -> ${targetPath}`)
    return ctx.redirect(targetPath)
  }

  // 2. 檢查重導向鏈長度（防止 Stack Overflow）
  if (ctx._redirectChain!.length >= 5) {
    console.warn(`重導向鏈過長: ${ctx._redirectChain!.length} 次，降級為外部重導向`)
    return ctx.redirect(targetPath)
  }

  // 3. 正常內部重導向
  // ... 執行重導向邏輯
}
```

## 🎯 系統架構

### 路徑正規化
```typescript
// 統一路徑處理，確保 /about 和 /about/ 匹配一致
getSpecialFilesChain(path: string): SpecialFiles[] {
  const normalizedPath = path === '/' ? '/' : path.replace(/\/$/, '');
  const parts = normalizedPath.split('/').filter(Boolean)
  
  // 從根開始收集特殊檔案
  const chain = [this.specialFiles.get('/') || {}]
  
  // 逐層收集
  let currentPath = ''
  for (const part of parts) {
    currentPath += '/' + part
    chain.push(this.specialFiles.get(currentPath) || {})
  }
  
  return chain
}
```

## 路由規則
- `index.tsx` → `/`
- `about.tsx` → `/about`
- `users/[id].tsx` → `/users/:id`
- `blog/posts/[slug].tsx` → `/blog/posts/:slug`

## 嵌套結構
- 支援巢狀特殊檔案（如 `/blog/_layout.tsx` 只影響 blog 路由）
- 中間件從根到目標路徑順序執行
- Layout 從外到內包裝

## 錯誤處理流程
1. **404 錯誤**：路由找不到時使用最近的 `_404.tsx`
2. **500 錯誤**：伺服器錯誤時使用最近的 `_500.tsx`
3. **一般錯誤**：其他錯誤使用最近的 `_error.tsx`

## 設計原則
1. **單一檔案**：所有路由邏輯在一個檔案中
2. **直接映射**：避免複雜的中間層
3. **Hono 整合**：直接返回 Hono Response
4. **錯誤處理**：簡單明瞭的錯誤流程
