# Project Specification: AI-Native Multi-tenant Website Generate (Deno + Hono)

## 1. 專案概述 (Project Overview)
建立一個基於 Deno 的高度動態、多租戶網站系統。
- **核心理念**：全資料庫驅動 (Database-driven)。路由、頁面、樣板均不依賴實體檔案，而是從資料庫讀取並動態渲染。
- **AI 介入**：除了系統預設組件外，頁面/內容/方塊(佈局樣板由) 皆由 AI 生成並以 JSON 或 HonoJSX 字串格式儲存。
- **解耦設計**：渲染層不直接操作 DB，完全透過 Hono 的內部調用 (`app.request`) 向 API 索取資料。

## 2. 技術棧 (Tech Stack)
- **Runtime**: Deno (使用最新穩定版)
- **Framework**: Hono (用於 API 與 JSX 伺服器端渲染)
- **Databases (三階段架構)**:
    1. **L1 (Deno KV)**: 儲存全域種子設定與系統資訊(系統設定)。
    2. **L2 (System SurrealDB)**: 儲存全網站共用資料（字典、圖示、預設樣板、佈景主題），以及網站設定（網站名稱、Host、描述、logo 等）。
    3. **L3 (Tenant SurrealDB)**: 儲存個別租戶（網站管理員）的私有內容與設定。
    - 資料庫會針對各種資料定義各種 model
    - model 會定義欄位的型態
    - seeds 會有各種同名model 的初始資料，middleware 在連線資料庫後會初始化資料庫決定是否把 seed資料加入資料庫！
    - **資料庫 ID 規則**：所有資料庫記錄的 ID 格式為 `table name:model name:uuid`
        - 例如：`系統資訊:系統資訊:預設`（系統預設資料使用固定 ID）
        - 例如：`系統資訊:標籤:321jsd3`（一般資料使用 UUID）
        - 此規則適用於 L2 (SurrealDB) 和 L3 (SurrealDB)，L1 (KV) 使用陣列格式作為 key
    - **連線資訊管理**：
        - L2 系統資料庫連線資訊存在 L1 (KV) 的系統資訊中，由系統管理員輸入（初始為空）。
        - L3 租戶資料庫連線資訊存在 L2 系統資料庫中，由網站管理員申請時輸入。
        - 不需要 `.env` 配置資料庫連線資訊。
- **環境變數 (.env)**:
    - `SECRET_KEY`: 密碼方法使用的金鑰。
    - `CACHE_ENABLED`: 快取開關（`true`/`false`），開發時建議關閉以免干擾 debug。
    - `AI_API_KEY`: LLM API 金鑰（用於 AI 樣板生成）。
    - `AI_PROVIDER`: LLM 提供商（如 `openai` / `anthropic`），預設 `openai`。
    - `LOG_LEVEL`: 日誌等級（`debug` / `info` / `warn` / `error`），預設 `info`。
    - `LOG_TO_FILE`: 是否將日誌寫入檔案（`true` / `false`），預設 `false`，生產環境建議開啟。
- **Styling**: 
    - UnoCSS (Runtime 模式)，支援 Tailwind 語法且無需掃描檔案。
    - 使用 @unocss/preset-wind4 套件
    - 使用自訂義 preset ，學習 daisyUI 的設計理念，定義顏色、字型、間距等 
        - 使用 css-variable 以快速切換佈景主題
        - 顏色使用 oklch 格式，用來計算文字顏色 (40%亮度以上黑色，以下白色)
    - **AI 整合**：提供 API 讓 AI 取得自訂 classes 列表，結合標準 TailwindCSS v4 使用

## 3. 核心架構規範 (Architectural Rules)

### A. 資料存取層 (API-First)
- 禁止在 Renderer 中直接引用資料庫 Client。
- 所有回傳都必須是 JSON 格式。
- 必須定義 `/api/v1/*` 路由處理所有 CRUD。
- 渲染層使用 `const res = await app.request('/api/v1/...')` 獲取數據，需在header傳入host/originalURL
- **共通欄位規範**：所有 Model 皆包含 `可刪除: boolean` 欄位。
    - Seed 資料預設 `可刪除 = false`，不可被刪除。
    - DELETE API 需檢查此欄位，若為 `false` 則回傳 `403 DELETE_PROTECTED` 錯誤。

### B. 統一資料庫訪問架構 (Three-Tier Database Architecture)
- **🚫 嚴格禁止直接資料庫訪問**：
    - 禁止在任何 Service、API、或 Middleware 中直接使用 `取得KV資料庫()`
    - 禁止直接建立 SurrealDB 連線或引用 SurrealDB Client
    - 禁止直接操作 L1 (KV)、L2 (System SurrealDB)、L3 (Tenant SurrealDB)
- **✅ 唯一允許的資料訪問方式**：
    - **三層查詢管理器** (`core/three-tier-query.ts`)：所有資料查詢必須透過此管理器
        - `三層查詢管理器.查詢單一<T>(c, model, id)` - 查詢單筆資料
        - `三層查詢管理器.查詢列表<T>(c, model, limit, offset)` - 查詢資料列表
        - `三層查詢管理器.創建或更新<T>(c, model, data)` - 創建或更新資料
        - `三層查詢管理器.取得預設值<T>(c, model)` - 取得預設值
    - **Context 預載資訊**：系統資訊和網站資訊透過 Middleware 預載到 Context
        - `c.get('系統資訊')` - 取得預載的系統資訊
        - `c.get('網站資訊')` - 取得預載的網站資訊
        - `c.get('kvDB')` - 僅限 Middleware 層設定，其他層禁止使用
    - **資料過濾器** (`utils/資料過濾器.ts`)：統一處理 API 回應資料
        - `資料過濾器.一般過濾(data, language)` - 處理單筆資料
        - `資料過濾器.列表過濾(data, language, mode)` - 處理資料列表
- **🔧 架構實現細節**：
    - **Middleware 層**：唯一可直接操作資料庫的層級
        - ✅ 可以使用 `c.get()` 和 `c.set()` 操作 Context
        - ✅ 可以直接使用 `三層查詢管理器` 查詢資料庫
        - ✅ 可以直接操作 KV 資料庫和 SurrealDB 連線
        - `middleware/db-resolver.ts` - 資料庫連線解析，設定 Context 中的資料庫實例
        - `middleware/info-loader.ts` - 預載系統資訊和網站資訊到 Context
        - `middleware/language-resolver.ts` - 語言解析，使用 Context 中的資訊
    - **API 層** (`services/apiService/*`)：業務邏輯處理層
        - ✅ 可以使用 `c.get()` 取得 Context 中的預載資訊
        - ✅ 可以使用 `三層查詢管理器` 進行資料查詢和操作
        - ❌ 禁止直接操作資料庫連線
        - ❌ 禁止直接使用 `取得KV資料庫()`
    - **Service 層** (`services/languageService/*` 等)：純業務邏輯層
        - ✅ 只能使用 `c.get()` 取得 Context 中的預載資訊
        - ❌ 禁止使用 `三層查詢管理器`（避免循環依賴）
        - ❌ 禁止直接查詢資料庫
        - ❌ 禁止直接操作資料庫連線
    - **Renderer 層** (`services/rendererService/*`)：渲染層
        - ✅ 只能透過 `app.request('/api/v1/...')` 調用 API
        - ❌ 禁止直接使用 Context
        - ❌ 禁止直接查詢資料庫
        - ❌ 禁止直接調用 Service
- **⚠️ 違規處理**：
    - KV 資料庫函數 (`取得KV資料庫()`) 已加入防護註解，明確標示禁止直接使用
    - 程式碼審查時必須檢查是否有違規的直接資料庫訪問
    - AI 編輯器必須遵循此架構限制，不得繞過三層查詢管理器
- theme/color/skeleton api 規則（更新為 KV seeds 邏輯）：
    - **預設值 API**（新增）：
        - `GET /api/v1/defaults` - 取得所有預設值（theme/color/skeleton）
        - `GET /api/v1/defaults/theme` - 取得預設佈景主題
        - `GET /api/v1/defaults/color` - 取得預設配色
        - `GET /api/v1/defaults/skeleton` - 取得預設骨架
        - 預設值來源：L3 → L2 → L1 (KV seeds)，保證有回傳值
        - 回傳格式包含 `isSystemSeed: true/false` 和 `source: "L1/L2/L3"`
    
    - **統一資訊 API**（基礎 API，其他 API 可能調用）：
        - `GET /api/v1/info` - 統一資訊端點
            - 優先返回網站資訊（如果存在）
            - 回退到系統資訊
            - 提供統一的語言列表來源
            - 回傳格式：
                ```json
                {
                  "success": true,
                  "data": {
                    "語言": ["zh-tw", "zh-cn", "en"],
                    "佈景主題": "theme-id",
                    "配色": "color-id",
                    "骨架": "skeleton-id"
                  },
                  "source": "website" | "system"
                }
                ```
        - `GET /api/v1/websites/info` - 網站資訊端點
        - `GET /api/v1/system/info` - 系統資訊端點

    - **特殊 API**（基於統一資訊的當前設定）：
        - `GET /api/v1/theme` - 取得當前網站主題（等同 `/api/v1/defaults/theme`）
        - `GET /api/v1/color` - 取得當前配色（等同 `/api/v1/defaults/color`）
        - `GET /api/v1/skeleton` - 取得當前骨架（等同 `/api/v1/defaults/skeleton`）
        - 所有 API 回傳包含 `success`, `data`, `source`, `error` 欄位
        - 系統 seeds 標示：`isSystemSeed: true` 且 `source: "L1"`

    - **完整 CRUD API**：
        - `GET /api/v1/themes` - 取得所有佈景主題
            - 三層查詢：L3 → L2 → L1 (KV seeds)
            - 支援 `?include_seeds=false` 參數過濾系統 seeds
            - 保證至少有系統 seeds 可回傳
        - `GET /api/v1/themes/:id` - 取得指定佈景主題
            - 三層查詢：L3 → L2 → L1
            - 保證系統 seeds 存在
        - `POST /api/v1/themes` - 建立新佈景主題
        - `PUT /api/v1/themes/:id` - 更新佈景主題
        - `DELETE /api/v1/themes/:id` - 刪除佈景主題
            - 檢查 `可刪除` 欄位，`false` 時禁止刪除
            - 系統 seeds（來自 L1 且 `可刪除: false`）無法刪除
            - 回傳 `SYSTEM_SEED_PROTECTED` 錯誤碼
    
    - **color / skeleton API**：
        - 比照 theme API 規格，支援完整 CRUD
        - 同樣的三層查詢邏輯和系統 seeds 保護
        - 路徑：`/api/v1/colors/*` 和 `/api/v1/skeletons/*`

    - **路由邏輯**：
        - 支援單數轉複數：`/api/v1/color` → `colors.ts`
        - 支援目錄模組：`/api/v1/websites/info` → `websites/info.ts`
        - 支援 action 參數：`/api/v1/defaults?action=theme`
- cube/page api 規則（更新為 KV seeds 邏輯）：
    - **pages API**：
        - `GET /api/v1/pages` - 取得所有頁面設定
            - 三層查詢：L3 → L2 → L1 (KV seeds)
            - 支援 `?include_seeds=false` 參數過濾系統 seeds
            - 回傳完整頁面資訊（不僅限 id/名稱/描述）
        - `GET /api/v1/pages/:id` - 取得指定頁面設定
            - 三層查詢：L3 → L2 → L1
            - 保證系統 seeds 存在
        - `GET /api/v1/pages/path/:path` - 取得符合路徑的頁面內容
            - 支援巢狀路徑查詢
            - 三層查詢邏輯
        - `POST /api/v1/pages` - 建立新頁面
        - `PUT /api/v1/pages/:id` - 更新頁面
        - `DELETE /api/v1/pages/:id` - 刪除頁面（檢查 `可刪除` 欄位）
    
    - **cubes API**：
        - `GET /api/v1/cubes` - 取得所有方塊設定
            - 三層查詢：L3 → L2 → L1 (KV seeds)
            - 支援 `?include_seeds=false` 參數
        - `GET /api/v1/cubes/:id` - 取得指定方塊
        - `POST /api/v1/cubes` - 建立新方塊
        - `PUT /api/v1/cubes/:id` - 更新方塊
        - `DELETE /api/v1/cubes/:id` - 刪除方塊（檢查 `可刪除` 欄位）

- **樣式資源 API**（新增）：
    - `GET /api/v1/styles/available` - 取得可用樣式資源
        - 回傳所有可用的 UnoCSS classes
        - 包含預設主題色彩變數
        - 包含可用組件類別（btn, card, input 等）
        - 格式：
            ```json
            {
              "success": true,
              "data": {
                "classes": ["btn", "btn-primary", "card", ...],
                "colors": ["primary", "secondary", "accent", ...],
                "components": ["button", "card", "input", ...],
                "themeVariables": {
                  "--primary": "oklch(...)",
                  "--secondary": "oklch(...)",
                  ...
                }
              }
            }
            ```
        - 取到 path 參數時，需在最前面加上"/"
        - 需處理巢狀路徑 如：api/v1/pages/path/abc/def => path = "/abc/def"
        - GET: 回傳指定的資源
        - POST: 建立新的資源
        - PUT: 更新資源
        - DELETE: 刪除資源，需檢查 `可刪除` 欄位，`false` 時禁止刪除
### B. 渲染管線 (Rendering Pipeline)
- **路由**：使用 `app.get('*')` 作為動態分發器。
    - 需依序攔截 api/ media 給對應的 apiService/mediaservice處理
    - 其餘路由給渲染器處理 renderService
    - 各自處理自己的錯誤及404
        - api: 返回 JSON 格式的錯誤訊息
        - render: 返回 404 頁面
        - media: 返回預設圖示
- **CSS 處理**：
    - 渲染 HTML 字串後，呼叫 `unocss.generate(html)`。
    - 將生成的 CSS 注入 `<head><style>`，實現真正的 On-demand CSS。
- **網站內容**：
    - 網站由骨架.佈局 取得最外增的佈局，佈局裡的 children 則填入網頁內容(依照路徑找到不同的內容)。
    - theme api/ skeletion api / color api 會回傳對應的資料
- **網頁內容**：
    - 網頁由 頁面 取得 佈局方塊(方塊id) 及 內容(內容 JSON)陣列。
    - 內容會指定使用的方塊及套用方塊的參數及內容
- **方塊類型**：
    1. **預設 (Hono Component)**: 實體檔案，動態 `import()`。
    2. **組合 (Composition)**: 資料庫紀錄的方塊組合，需支援巢狀結構。(可限制深度)
    3. **AI (JSX String)**: 資料庫儲存的 JSX 字串，透過渲染器解析執行。

### C. 多國語言 (MultiLanguage)
- 使用 jsr:@dui/smartmultilingual@0.2.2 套件
- **語言服務架構**：
  - 建立 `services/languageService` 進行統一語言管理
  - 語言支援列表動態從統一資訊 API 取得
  - 自動驗證請求語言是否支援，不支援時回退到預設語言

- **語言解析優先順序**：
  1. **Cookie 設定的語言**
     - Cookie 名稱：`lang`
     - 路徑：`/`
  2. **瀏覽器設定的語言**（Accept-Language 標頭）
     - 解析 Accept-Language 標頭，例如：`zh-TW,zh;q=0.9,en;q=0.8`
     - 尋找支援的語言並驗證
  3. **統一資訊 API 語言列表**
     - 從 `/api/v1/info` 取得支援的語言陣列
     - 使用陣列中的第一個語言
  4. **預設語言**：`zh-tw`

- **統一資訊 API**：
  - 端點：`/api/v1/info`
  - 優先返回網站資訊（如果存在）
  - 回退到系統資訊
  - 提供統一的語言列表來源

- **API 語言使用**：
  - 語言設定儲存在 context：`c.set('語言', language)`
  - API 使用 `c.get('語言')` 獲取當前語言
  - 所有 API 模組使用統一資訊 API 取得設定
  - 多國語言字串使用 `toStringAsync(language)` 處理

- **路由邏輯**：
  - 支援單數轉複數：`/api/v1/color` → `colors.ts`
  - 支援目錄模組：`/api/v1/websites/info` → `websites/info.ts`
  - 移除路徑語言處理邏輯

- **資料庫 Model**：
  - 使用 MultilingualString/SmartContent/MultilingualSmartContent 儲存多國語言內容
  - API 回應自動根據語言設定過濾內容

### D. 內部 API 調用規範 (Internal API Calling)

#### D.1. Service 層 API 調用
- **🔧 強制使用 InnerAPI**：
  - 所有 Service 層調用 API 必須使用 `services/index.ts` 中的 `InnerAPI` 函數
  - **禁止**直接使用 `app.request()` 或其他 HTTP 客戶端
  - **禁止**在各個 API 檔案中重複定義 InnerAPI 函數
- **InnerAPI 功能**：
  - 自動編碼 URL 參數（如 `?route=/en/home` → `?route=%2Fen%2Fhome`）
  - 統一錯誤處理和日誌記錄
  - 自動添加必要的 HTTP 標頭（host、origin）
- **使用方式**：
  ```typescript
  import { InnerAPI } from '../../services/index.ts';
  
  // 正確方式
  const response = await InnerAPI(c, '/api/v1/page?route=/en/home');
  
  // 錯誤方式（禁止）
  const response = await app.request('/api/v1/page?route=/en/home');
  ```

#### D.2. API 方法擴充規範
- **🔧 簡化的 API 方法體系**：
  - **GET**：取得資源（支援所有 query string 參數）
  - **POST**：新增資源
  - **PUT**：更新資源（支援 query string 參數指定特定資源）
  - **DELETE**：刪除資源（支援 query string 參數指定特定資源）
  - **CURRENT**：取得當前資源（取代原來的 ONE 方法）
- **GET 方法的智能路由**：
  - **無參數**：`GET /api/v1/color` → CURRENT 方法（當前資源）
  - **有參數**：`GET /api/v1/color?id=xxx` → GET 方法（處理所有參數）
  - **特殊參數**：`GET /api/v1/color?all=true` → GET 方法內部處理（全部資源）
  - **路徑參數**：`GET /api/v1/page?path=/en/home` → GET 方法內部處理（頁面路徑）
- **參數傳遞方式**：
  - 所有參數使用 query string 格式：`?id=xxx&path=/en/home&all=true`
  - **不再使用路徑參數**：如 `/api/v1/colors/123` 改為 `/api/v1/color?id=123`
  - **自動 URL 解碼**：所有參數使用 `decodeURIComponent()` 解碼
- **方法對應範例**：
  ```typescript
  // 原來的方式
  GET /api/v1/colors/123 → colors.ts GET(params.id = "123")
  ONE /api/v1/color → colors.ts ONE()
  
  // 新的方式
  GET /api/v1/color (無參數) → color.ts CURRENT()
  GET /api/v1/color?id=配色:配色:經典藍 → color.ts GET()
  GET /api/v1/color?all=true → color.ts GET() 內部處理全部
  GET /api/v1/page?route=/en/home → page.ts GET() 內部處理路徑
  ```
- **檔案命名統一**：
  - 全部使用單數命名：`color.ts`、`theme.ts`、`skeleton.ts`、`page.ts`
  - 不再使用複數檔案：`colors.ts`、`themes.ts` 等

#### D.3. API 層參數接收
- **🔧 強制參數解碼**：
  - 所有 API 接收 query string 參數後必須使用 `decodeURIComponent()` 解碼
  - **特別注意**包含 `/`、`?`、`&` 等特殊字符的參數
- **解碼規則**：
  ```typescript
  // 正確方式
  export async function GET(c: Context) {
    const route = c.req.query('route');
    const decodedRoute = route ? decodeURIComponent(route) : undefined;
    
    if (decodedRoute === "/en/home") {
      // 處理解碼後的路徑
    }
  }
  
  // 錯誤方式（可能導致路徑錯誤）
  const route = c.req.query('route'); // 直接使用未解碼的值
  ```
- **常見需要解碼的參數**：
  - `route` - 頁面路徑（如 `/en/home`）
  - `path` - 檔案路徑（如 `/images/logo.png`）
  - `url` - 完整 URL（如 `https://example.com/page`）
  - 任何包含 `/` 的參數

#### D.4. 編碼/解碼流程
1. **調用方**：使用 InnerAPI，自動編碼參數
2. **傳輸**：URL 參數已編碼（如 `%2Fen%2Fhome`）
3. **接收方**：使用 decodeURIComponent 解碼參數
4. **處理**：使用解碼後的原始值進行業務邏輯

#### D.5. 違規處理
- **程式碼審查**：必須檢查是否有直接使用 `app.request()` 的情況
- **重複定義檢查**：禁止在各個檔案中重複定義 InnerAPI 函數
- **參數解碼檢查**：必須檢查 API 是否正確解碼 query string 參數
- **AI 編輯器限制**：AI 必須遵循此規範，不得繞過 InnerAPI

### E. 認證/授權系統 (Authentication/Authorization)
- **第一階段**：暫不實作認證機制，所有 API 開放存取。
- **後續階段**：加入「使用者」與「角色」兩個 Model，POST/PUT/DELETE 需認證保護。

### F. Media Service 規格
- **路由攔截**：`/medias/*` 由 mediaService 處理。
- **路徑格式**：
    - 列表：`/medias/{type}` — 取得該類型所有媒體資源列表（JSON）。
    - 單筆：`/medias/{type}/:id` — 取得指定媒體資源（回傳媒體本身）。
- **列表查詢參數**（適用於 `/medias/{type}`）：
    | 參數 | 說明 | 預設值 |
    |------|------|--------|
    | `page` | 頁碼 | `1` |
    | `pageSize` | 每頁筆數 | `20` |
    | `sort` | 排序欄位（如 `name`, `createdAt`） | `createdAt` |
    | `order` | 排序方向（`asc` / `desc`） | `desc` |
    | `search` | 關鍵字搜尋（名稱/描述） | — |
    - 列表回傳格式遵循 §3.F 標準，`data` 中額外包含分頁資訊：
    ```json
    {
        "success": true,
        "data": {
            "items": [ ... ],
            "pagination": {
                "page": 1,
                "pageSize": 20,
                "total": 100,
                "totalPages": 5
            }
        },
        "source": "system"
    }
    ```
- **媒體類型與預設回傳**：
    | 路徑 | 說明 | 找不到時的預設回傳 |
    |------|------|--------------------|
    | `/medias/icons/:id` | 圖示 | 預設 SVG 圖示 |
    | `/medias/images/:id` | 圖片 | 預設佔位圖片 (placeholder) |
    | `/medias/videos/:id` | 影片 | 預設影片封面或錯誤訊息 |
    | `/medias/scripts/:id` | 腳本 | 空 JS 或錯誤訊息 |
- **特殊路徑**：
    - `/medias/icon` → 抓取網站資訊/系統資訊的「商標」欄位，沒有則傳回預設圖示（作為 favicon 使用）。
- **儲存策略**：
    - SVG 圖示存儲在 SurrealDB。
    - 其他媒體儲存方式待定，預計未來支援 MinIO。
- **錯誤處理**：找不到媒體時回傳對應類型的預設資源（而非 JSON 錯誤）。

### G. API 回應格式標準 (API Response Format)
- **成功回應**：
    ```json
    {
        "success": true,
        "data": { ... },
        "source": "site" | "system" | "default"
    }
    ```
- **錯誤回應**：
    ```json
    {
        "success": false,
        "error": {
            "code": "NOT_FOUND",
            "message": "找不到指定的資源",
            "status": 404
        }
    }
    ```
- **HTTP Status Code 規範**：
    | Status | 用途 |
    |--------|------|
    | 200 | 查詢/更新成功 |
    | 201 | 新增成功 |
    | 400 | 請求格式錯誤（缺少必要欄位等） |
    | 403 | 禁止操作（如刪除 `可刪除=false` 的資料） |
    | 404 | 資源不存在 |
    | 422 | 資料驗證失敗（格式正確但內容不合規） |
    | 500 | 伺服器內部錯誤 |
- **錯誤 Code 常數**：`NOT_FOUND` / `FORBIDDEN` / `DELETE_PROTECTED` / `VALIDATION_ERROR` / `BAD_REQUEST` / `INTERNAL_ERROR`

### K. AI 方塊生成規範 (AI Cube Generation)

#### 1. API 端點
- `GET /api/v1/styles/classes`：回傳自訂 preset 的可用 class 列表，供 AI 生成時參考。
- `POST /api/v1/ai/generate-cube`：接收使用者描述，產生方塊樣板。
    - 請求 Body：`{ "描述": "一個藍色色調的英雄產品區", "目標": "system" | "tenant" }`
    - 回應：遵循 §3.F 格式，`data` 包含生成的方塊資料（含 Hono JSX 字串）。

#### 2. LLM Provider
- 支援 GPT-4o / Claude / ollama 等主流 LLM。
- API Key 透過 `.env` 的 `AI_API_KEY` 配置。
- 未來可擴充支援多個 Provider，透過 `.env` 的 `AI_PROVIDER` 切換（如 `openai` / `anthropic` / `ollama`）。

#### 3. Prompt 約束 (Prompt Engine)
- 在 `utils/ai-prompts.ts` 定義標準 System Prompt，強制以下規則：
    - **僅使用 Hono JSX 語法**（非 React JSX）。
    - **優先使用自訂 preset class**（透過 `/api/v1/styles/classes` 動態取得，注入至 System Prompt）。
    - **可搭配 Tailwind 原生 utility class**（UnoCSS 相容）。
    - **禁止 arbitrary values**（如 `text-[#ff0000]`），因系統有自訂色彩系統。
    - **禁止** `<script>` 標籤。
    - **禁止** inline event handlers（如 `onclick`、`onmouseover`）。
    - **組件必須接收 `props`** 作為資料輸入，保持資料與樣板分離。

#### 4. 安全管線 (AI-to-DB Pipeline)
- AI 生成流程：
    ```
    取得可用 class 列表 → 注入 System Prompt → LLM 生成 → 安全過濾 → 語法驗證 → 存入 DB
    ```
- **安全過濾** (`utils/sanitizer.ts`)：
    - 使用 `deno-dom` 解析 HTML/JSX 字串。
    - 移除所有 JavaScript 執行路徑（`<script>`、`javascript:` URI、inline event handlers）。
    - 僅保留 HTML 結構與 UnoCSS 相容的 class 屬性。
- **語法驗證**：
    - 檢查 JSX 字串是否能被正常解析。
    - 驗證失敗則拒絕寫入資料庫，回傳 `422 VALIDATION_ERROR`。

#### 5. JSX 字串渲染
- Renderer 從資料庫取得 AI 方塊時：
    - 將 JSX 字串透過渲染器解析為可渲染組件。
    - 渲染前再次經過安全過濾，防止資料庫被直接篡改的情況。
    - 解析失敗時顯示錯誤佔位元件（而非導致伺服器崩潰）。

#### 6. 失敗處理
- **LLM 呼叫失敗**：回傳 `500 INTERNAL_ERROR`，附帶錯誤描述。
- **產出不合法**：回傳 `422 VALIDATION_ERROR`，可選擇自動重試一次。
- **渲染失敗**：顯示錯誤佔位元件，記錄日誌，不影響其他方塊。

### K. 日誌系統 (Logging)
- **統一日誌介面**：在 `utils/logger.ts` 建立統一的 Logger，所有模組透過此介面輸出日誌，禁止直接使用 `console.log`。
- **日誌等級**（透過 `.env` 的 `LOG_LEVEL` 控制）：
    | 等級 | 用途 |
    |------|------|
    | `debug` | 開發除錯用，詳細資訊（API 請求參數、DB 查詢等） |
    | `info` | 一般運行資訊（伺服器啟動、請求處理、DB 連線等） |
    | `warn` | 警告（效能問題、棄用 API 呼叫、fallback 觸發等） |
    | `error` | 錯誤（未捕獲異常、DB 連線失敗、AI 生成失敗等） |
- **輸出方式**：
    - 預設輸出至 console（開發用）。
    - `LOG_TO_FILE=true` 時同時寫入檔案（存放於 `/logs` 目錄），生產環境使用。
- **日誌格式**：`[YYYY-MM-DD HH:mm:ss] [LEVEL] [module] message`
- **各 Service 日誌重點**：
    - **apiService**：請求方法、路徑、狀態碼、回應時間。
    - **mediaService**：媒體存取、預設回傳觸發。
    - **rendererService**：頁面渲染時間、方塊載入錯誤。
    - **AI 生成**：LLM 呼叫時間、成功/失敗、過濾結果。

## 4. 目錄結構 (Folder Structure)
- `/core`: 數據庫連線初始化與 UnoCSS 配置。
- `/data`: KV資料庫實體檔案位置。
- `/database`: 資料庫相關檔案。
    - `/models`: 資料庫模型檔案。
    - `/seeds`: 資料庫初始資料。
- `/docs`: 各種文件位置。
- `/middleware`: DB 解析器 (Tenant Resolver)、UnoCSS 注入器、語言解析器 (Language Resolver)。
- `/services`:
    - `/apiService`: 負責 L1/L2/L3 的資料邏輯。 攔截路由器的 /api/* 路由。
        - 支援統一資訊 API 和動態語言驗證
        - 包含 `websites/info.ts`、`info.ts` 等統一資訊端點
    - `/mediaService`: 負責媒體檔案服務。 攔截路由器的 /medias/* 路由。

        - 從統一資訊 API 取得支援語言列表
        - 語言驗證和回退機制
    - 錯誤處理：所有Service 都需要有錯誤處理機制，並自己處理自己的 404 錯誤。
- `/components`: 存放預設的方塊組件 .tsx 組件。
- `/utils`: 安全過濾 (Sanitize)、JSX 字串解析器。
- `main.ts`: 程式進入點。
- `.windsurfrules`: AI 協作規範。
